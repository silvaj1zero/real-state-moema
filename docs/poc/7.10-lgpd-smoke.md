## Fluxo end-to-end (6 etapas)

### Etapa 1 — Criar lead simulando scrape

Simula um crawler PF (Story 7.4) inserindo um lead. Não toca PII ainda — só metadata.

```sql
-- Como service_role
INSERT INTO leads (
  id, source, consultant_id, lead_type, origem, status
) VALUES (
  '00000000-0000-0000-0000-000000000001',
  'fisbo_zap_test',
  '<uuid_consultor_de_teste>',
  'FISBO',
  'scrape_test',
  'novo'
)
RETURNING id;
```

**Estado esperado:** 1 row em `leads`; `telefone_secret_id`, `email_secret_id`, `whatsapp_secret_id`, `nome_secret_id` todos `NULL`; `lgpd_status = 'active'`.

---

### Etapa 2 — Cifrar PII via wrapper `lib/vault.ts`

Em TS, exercita `storeLeadPII` (RPC `fn_store_lead_pii`):

```ts
import { createClient } from '@supabase/supabase-js'
import { storeLeadPIIBatch } from '@/lib/vault'

const supabase = createClient(SUPABASE_URL, SERVICE_ROLE_KEY)

await storeLeadPIIBatch(supabase, '00000000-0000-0000-0000-000000000001', {
  telefone: '11999998888',
  email: 'fulano@example.com',
  nome: 'Fulano de Tal',
})
```

**Estado esperado:**

```sql
-- leads tem secret_ids populados
SELECT telefone_secret_id, email_secret_id, nome_secret_id, whatsapp_secret_id
FROM leads WHERE id = '00000000-0000-0000-0000-000000000001';
-- → 3 UUIDs não-null + 1 NULL (whatsapp não passado)

-- vault.secrets tem 3 rows com names canônicos
SELECT name FROM vault.secrets
WHERE name LIKE 'lead_%_00000000-0000-0000-0000-000000000001'
ORDER BY name;
-- → lead_email_..., lead_nome_..., lead_telefone_...
```

Auditoria automática (trigger / RPC): NÃO há row em `lgpd_audit_log` para store (apenas reveal/scrape/anonymize geram audit). Store é etapa de cifragem — confiança vem do trigger de scrape (Etapa 1b abaixo, opcional via `fn_lgpd_log_scrape`).

### Etapa 2b — Log de scrape (action='scrape')

```ts
import { logLGPDScrape } from '@/lib/lgpd-audit'

await logLGPDScrape(

```

**Estado esperado:**

```sql
SELECT action, legal_basis, evidence
FROM lgpd_audit_log
WHERE lead_id = '00000000-0000-0000-0000-000000000001';
-- → 1 row: action='scrape', legal_basis='legitimate_interest', evidence={'source':'fisbo_zap_test','portal':'zap'}
```

### Etapa 3 — Reveal-on-demand (gera audit)

Consultor abre o lead na UI → backend chama `fn_decrypt_lead_pii` via wrapper:

```ts
import { getLeadPII } from '@/lib/vault'

const phone = await getLeadPII(supabase, '00000000-0000-0000-0000-000000000001', 'telefone')
console.log(phone) // '11999998888'
```

**Estado esperado:** após a chamada:

```sql
SELECT action, legal_basis, evidence
FROM lgpd_audit_log
WHERE lead_id = '00000000-0000-0000-0000-000000000001'
ORDER BY timestamp DESC LIMIT 2;
-- → row mais recente: action='reveal_phone', legal_basis='legitimate_interest',
--   evidence inclui {'field':'telefone'}
```

Edge case a confirmar: `getLeadPII(..., 'whatsapp')` retorna `null` (nunca foi capturado) E **não** gera audit row de reveal (RPC já trata short-circuit antes de logar). Validar em execução real.

### Etapa 4 — Opt-out request (titular submete)

Simula o titular preenchendo o form em `/lgpd/opt-out`:

```bash
curl -X POST http://localhost:3000/api/lgpd/opt-out \
  -H 'Content-Type: application/json' \
  -d '{
    "telefone": "(11) 99999-8888",
    "evidence": "Não autorizei contato — recebi ligação em 2026-05-17"
  }'
```

**Estado esperado:**

```json
HTTP 202 Accepted
{
  "protocol_number": "OPT-2026-05-18-XXXX",
  "status": "pending",
  "sla_days": 15,
  "message": "Solicitação registrada..."
}
```

E no banco:

```sql
SELECT protocol_number, telefone, status FROM lgpd_opt_out_requests
ORDER BY created_at DESC LIMIT 1;
-- → 'OPT-2026-05-18-XXXX', '11999998888', 'pending'

SELECT action, evidence FROM lgpd_audit_log
WHERE evidence->>'protocol_number' = 'OPT-2026-05-18-XXXX';
-- → action='opt_out_request', legal_basis='legal_obligation'
```

### Etapa 5 — Admin processa manualmente

Admin loga no `/admin/lgpd`, clica "processar" → backend chama RPC `fn_lgpd_process_opt_out`:

```bash

curl -X POST http://localhost:3000/api/admin/lgpd/process-opt-out \
  -H 'Authorization: Bearer <admin_jwt>' \
  -H 'Content-Type: application/json' \
```

**Estado esperado:**

```json

  "protocol_number": "OPT-2026-05-18-XXXX",
  "matched": 1,
```

### Etapa 6 — Assert anonimização

Após processamento:

```sql
-- A) Lead anonimizado
SELECT lgpd_status, lgpd_anonymized_at,
       telefone_secret_id, email_secret_id, nome_secret_id
FROM leads WHERE id = '00000000-0000-0000-0000-000000000001';
-- → lgpd_status='anonymized', lgpd_anonymized_at=now,
--   *_secret_id todos NULL

-- B) Vault secrets PURGED (não basta apenas zerar FK; secret tem que sumir)
SELECT count(*) FROM vault.secrets
WHERE name LIKE 'lead_%_00000000-0000-0000-0000-000000000001';
-- → 0

-- C) Reveal pós-anonymize falha
-- (em TS: getLeadPII(...'telefone') joga "lead opted out" OR retorna null)

-- D) Opt-out request concluída
SELECT status, processed_at FROM lgpd_opt_out_requests
WHERE protocol_number = 'OPT-2026-05-18-XXXX';
-- → status='completed', processed_at=now

-- E) Audit trail completo
SELECT action, legal_basis, timestamp
FROM lgpd_audit_log
WHERE lead_id = '00000000-0000-0000-0000-000000000001'
   OR evidence->>'protocol_number' = 'OPT-2026-05-18-XXXX'
ORDER BY timestamp;
-- → scrape, reveal_phone, opt_out_request, anonymize, opt_out_complete (5 rows)

-- F) NOTA — `scraped_listings` na Wave A NÃO armazena PII clara (a coluna foi
--    movida para `leads.*_secret_id`). O AC11 original menciona "assert NULL em
--    scraped_listings" — interpretação atual: a tabela `scraped_listings` da
--    Story 6.1 não recebe PII; o assert equivalente é (A) + (B) acima.
```

## Edge cases a validar em execução real

1. **Idempotência store:** chamar `storeLeadPII` 2x no mesmo `(lead_id, field)` deve **atualizar** o secret existente (não criar duplicata). RPC já trata via `UPDATE` se `vault.secrets.name` colide.
2. **Reveal de campo nunca capturado:** `getLeadPII(..., 'whatsapp')` → retorna `null`, sem erro, sem audit row de reveal.
3. **RLS bridge:** authenticated com `auth.uid()` diferente de `lead.consultant_id` → `fn_decrypt_lead_pii` RAISE EXCEPTION.
4. **Opt-out sem match:** protocol válido mas nenhum lead match → `matched=0`, request marcado `completed` ainda assim (compliance: titular foi atendido).
5. **Vault delete contra constraint:** se algum FK ainda aponta para secret deletado → `ON DELETE SET NULL` deve evitar erro.
6. **Retention 90d:** Edge Function cron (dependência da Story 7.6) deve executar `fn_lgpd_purge_expired_audit()` — não há trigger DB, é cron-only.
