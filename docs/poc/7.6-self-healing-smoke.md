# PoC 7.6 — Self-Healing Smoke (E2E, doc-only)

**Date:** 2026-05-18
**Story:** 7.6 — Pipeline cron-Supabase + Self-Healing
**AC:** AC9 — smoke E2E validates pgmq + cron + fn_mark_stale_runs + fn_retry_failed_runs end-to-end.
**Mode:** Documented procedure (executes on staging Supabase branch after migrations applied).
**Author:** Dex (@dev)

---

## Goal

Prove that the self-healing pipeline correctly:

1. Marks a run stuck in `running` for >15min as `failed` (AC2).
2. Re-enqueues that failed run via `pgmq` and increments `retry_count` (AC3).
3. After 3 retries, leaves the run in `failed` permanently — no infinite loop (AC3 cap).

---

## Prerequisites

- Migrations 012 + 013 applied (staging branch).
- `pgmq` 1.5.1 + `pg_cron` 1.6.4 installed (verified via `docs/poc/7.6-pgmq-availability-check.md`).
- `epic7_stale_runs_healer` cron job present in `cron.job`.
- Service role key + Edge Function base URL set via `app.service_role_key` / `app.edge_function_base_url` PG settings.

## Step 1 — Seed a stale run

```sql
INSERT INTO crawl_runs (portal, status, started_at, retry_count)
VALUES ('mercadolivre', 'running', NOW() - INTERVAL '16 minutes', 0)
RETURNING id;
-- save returned id as :run_id
```

## Step 2 — Trigger healer manually (skip cron wait)

```sql
SELECT fn_mark_stale_runs();
-- expected: NOTICE "marked 1 stale runs as failed"
-- expected: return 1
```

## Step 3 — Verify failed state

```sql
SELECT status, error_message, finished_at
FROM crawl_runs WHERE id = :run_id;
-- expected:
--   status = 'failed'
--   error_message LIKE 'stale: started_at=%'
--   finished_at IS NOT NULL
```

## Step 4 — Trigger retry round 1

```sql
SELECT fn_retry_failed_runs(3);
-- expected: NOTICE "re-enqueued 1 failed runs (max_retries=3)"
-- expected: return 1
```

## Step 5 — Verify pgmq enqueue + retry_count increment

```sql
SELECT retry_count FROM crawl_runs WHERE id = :run_id;
-- expected: 1

SELECT * FROM pgmq.read('epic7_long_tasks', vt := 5, qty := 10);
-- expected: 1 message with body:
--   { "kind": "retry_crawl_run", "run_id": "<:run_id>", "portal": "mercadolivre", "retry_count": 1 }
```

## Step 6 — Retry rounds 2 and 3

Re-run Step 4 two more times. After each:

- `retry_count` should be 2, then 3.
- New pgmq message per round.

## Step 7 — Verify cap at max_retries

```sql
SELECT fn_retry_failed_runs(3);
-- expected: NOTICE NOT emitted (count = 0)
-- expected: return 0
```

retry_count = 3 == p_max_retries, so the row is no longer eligible. The run
stays in `failed` permanently — anti-infinite-loop guarantee.

## Step 8 — Verify v_crawl_health (AC10)

```sql
SELECT * FROM v_crawl_health WHERE portal = 'mercadolivre';
-- expected columns: total_runs_24h, completed, failed, running,
--   cancelled, success_rate_pct, avg_duration_ms, total_retries, last_run_at
-- expected: failed >= 1, total_retries = 3
```

---

## Cleanup

```sql
DELETE FROM crawl_runs WHERE id = :run_id;
SELECT pgmq.purge_queue('epic7_long_tasks');
```

## Local execution guide (QA fix TEST-001)

This section closes the gate finding that AC9 had no reproducible-locally path.
@qa (Quinn) can execute the full self-healing loop against a local Supabase
instance without waiting on staging deploy.

### 0. Prerequisites

- Supabase CLI installed (`supabase --version` >= 1.200).
- Docker running (Supabase local stack needs it).
- Postgres client on PATH (`psql --version`).
- Repo working tree clean OR migrations 008-019 already merged to working
  branch (Story 7.6 fixes include 013 in-place edit + 018 + 019).

### 1. Start local Supabase + apply migrations 008-019

```bash
cd "C:/Users/Zero/Desktop/Real State - Moema"
supabase start                 # boots postgres, kong, studio, edge runtime
supabase db reset              # drops + re-applies ALL migrations in order
```

`supabase db reset` re-runs the full migration chain — migrations 012, 013,
018, 019 will execute in timestamp order. Watch the output for:

- `NOTICE: [migration 019] cron.timezone = UTC` (FIX 3 happy path) OR a
  WARNING if local Supabase has a non-UTC timezone (still applies; smoke
  proceeds).
- `CREATE EXTENSION pgmq VERSION '1.5.1'` succeeds (FIX 1).
- `ALTER VIEW public.v_crawl_health SET (security_invoker = true)`
  succeeds (FIX 2).

If any of the above fails, halt and route to @data-engineer.

### 2. Open a psql session to local Postgres

```bash
psql "postgresql://postgres:postgres@127.0.0.1:54322/postgres"
```

Port 54322 is the Supabase CLI default; confirm with `supabase status`.

### 3. Seed fixture run (accelerated stale)

Instead of waiting 16 real minutes, back-date `started_at`:

```sql
-- Inject a fixture run that is already "stale" by construction
INSERT INTO crawl_runs (portal, status, started_at, retry_count)
VALUES ('mercadolivre', 'running', NOW() - INTERVAL '16 minutes', 0)
RETURNING id;
-- Capture the returned UUID for the next steps.
-- Bash convenience: \gset run_  then reference :run_id
```

Alternative if you want to test the time-based trigger end-to-end (slower):

```sql
-- Insert at NOW(), then UPDATE to back-date it
INSERT INTO crawl_runs (portal, status, started_at, retry_count)
VALUES ('mercadolivre', 'running', NOW(), 0)
RETURNING id; -- :run_id

-- Wait some seconds, then back-date
UPDATE crawl_runs
   SET started_at = NOW() - INTERVAL '16 minutes'
 WHERE id = :'run_id';
```

### 4. Trigger healer manually (do NOT wait for the 5-min cron)

```sql
SELECT fn_mark_stale_runs();
-- Expect: NOTICE '[fn_mark_stale_runs] marked 1 stale runs as failed'
-- Expect: returns 1
```

Verify state transition:

```sql
SELECT id, status, error_message, finished_at, retry_count
  FROM crawl_runs
 WHERE id = :'run_id';
-- Expect:
--   status = 'failed'
--   error_message LIKE 'stale: started_at=%'
--   finished_at IS NOT NULL
--   retry_count = 0
```

### 5. Test retry cap (3 iterations + cap assertion)

Run `fn_retry_failed_runs(3)` repeatedly and assert `retry_count` increments
up to 3, then stops:

```sql
-- Iteration 1
SELECT fn_retry_failed_runs(3);                        -- returns 1
SELECT retry_count FROM crawl_runs WHERE id = :'run_id';  -- expect 1

-- Iteration 2
SELECT fn_retry_failed_runs(3);                        -- returns 1
SELECT retry_count FROM crawl_runs WHERE id = :'run_id';  -- expect 2

-- Iteration 3
SELECT fn_retry_failed_runs(3);                        -- returns 1
SELECT retry_count FROM crawl_runs WHERE id = :'run_id';  -- expect 3

-- Iteration 4 — must NO-OP (cap reached)
SELECT fn_retry_failed_runs(3);                        -- returns 0
SELECT retry_count FROM crawl_runs WHERE id = :'run_id';  -- still 3
```

Anti-infinite-loop guarantee: iteration 4 returns 0 because the row no
longer matches `retry_count < p_max_retries`.

### 6. Verify pgmq enqueue

Each successful retry pushed a message to `epic7_long_tasks`:

```sql
SELECT msg_id, message
  FROM pgmq.read('epic7_long_tasks', vt := 5, qty := 10);
-- Expect 3 messages, each:
--   { "kind": "retry_crawl_run", "run_id": "<uuid>",
--     "portal": "mercadolivre", "retry_count": 1|2|3 }
```

### 7. Verify v_crawl_health (AC10)

```sql
SELECT * FROM v_crawl_health WHERE portal = 'mercadolivre';
-- Expect: failed >= 1, total_retries = 3, success_rate_pct = 0 (or NULL)
```

If you are connected as service_role you see the row. If you connect as
`authenticated` (psql with the anon JWT) and crawl_runs has no permissive
RLS policy for that role, you get zero rows — that is the SEC-001 fixed
behavior (security_invoker now declared explicitly in migration 018).

### 8. Cleanup local fixture

```sql
DELETE FROM crawl_runs WHERE id = :'run_id';
SELECT pgmq.purge_queue('epic7_long_tasks');
```

Then either `supabase stop` (preserves DB) or `supabase db reset` (wipe).
