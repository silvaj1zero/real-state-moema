## Decision

**Path A — Use Supabase Vault natively for Epic 7 sensitive data (RECOMMENDED).**

Rationale:
1. Already installed — zero infra cost.
2. Decryption key is managed by Supabase Auth, never exposed to application code.
3. Auditable: `vault.secrets` has `created_at`/`updated_at`; access goes through a view that respects role grants.
4. Avoids `current_setting('app.encryption_key')` anti-pattern (key-in-session-var leaks via logs).

### Usage examples

**Path A — Supabase Vault (selected):**

```sql
-- Store a per-lead encrypted phone number (one secret per row pattern)
SELECT vault.create_secret(
  '11999998888',
  'lead_phone_' || lead_id::text,
  'WhatsApp do lead capturado via FISBO'
);

-- Read it back (only roles with grant on the view can decrypt)
SELECT decrypted_secret
FROM vault.decrypted_secrets
WHERE name = 'lead_phone_' || $1::text;
```

For high-volume lead PII, prefer storing a `secret_id uuid` foreign key on the `leads` table and reference `vault.decrypted_secrets.id` — this avoids one-secret-per-row name collisions and lets you `LEFT JOIN` decrypt only when authorized.

```sql
ALTER TABLE leads
  ADD COLUMN telefone_secret_id uuid REFERENCES vault.secrets(id);

-- Read pattern (server-side only, RLS gates access to lead row first)
SELECT l.id, l.nome, ds.decrypted_secret AS telefone
FROM leads l
LEFT JOIN vault.decrypted_secrets ds ON ds.id = l.telefone_secret_id;
```

**Path B — `pgcrypto` (fallback, NOT selected):**

```sql
-- Symmetric encryption with a session-scoped key (KEY MANAGEMENT IS YOUR PROBLEM)
INSERT INTO leads (telefone_encrypted)
VALUES (pgp_sym_encrypt('11999998888', current_setting('app.encryption_key')));

SELECT pgp_sym_decrypt(telefone_encrypted, current_setting('app.encryption_key'))
FROM leads;
```

Path B is documented for completeness but introduces key-rotation pain and risks leaking the key via `pg_stat_statements` / logs. Use only if Vault becomes unavailable.
