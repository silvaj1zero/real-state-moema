## Why deferred is acceptable

The filter is the **only** Epic-7-specific logic in this story. Upstream
scripts (`dados_cnpj_baixa.py`, `dados_cnpj_para_sqlite.py`) are
unmodified from rictom/cnpj-sqlite — their behavior is governed by the
pinned SHA. The push script is a thin SQL wrapper around standard
`psql \copy` + `INSERT ... ON CONFLICT`. Risk surface for the deferred
items is therefore bounded to: (a) RFB share_token expiry, (b) upstream
SHA drift, (c) Supabase auth — all already documented in `infrastructure/
cnpj-etl/README.md` Troubleshooting.

## How to run the full real-data smoke (for @devops)

```bash
# 1. Materialize upstream scripts (host with internet)
cd infrastructure/cnpj-etl
RICTOM_SHA=$(grep '^ARG RICTOM_SHA=' Dockerfile | cut -d= -f2)
curl -fsSL "https://raw.githubusercontent.com/rictom/cnpj-sqlite/${RICTOM_SHA}/dados_cnpj_baixa.py"      > scripts-upstream/dados_cnpj_baixa.py
curl -fsSL "https://raw.githubusercontent.com/rictom/cnpj-sqlite/${RICTOM_SHA}/dados_cnpj_para_sqlite.py" > scripts-upstream/dados_cnpj_para_sqlite.py

# 2. Configure .env
cat > .env <<EOF
DATABASE_URL=$STAGING_DATABASE_URL
EOF

# 3. Apply migration 011 to staging Supabase
supabase db push --linked --workdir ../..

# 4. Run pipeline (host needs ≥40 GB free, ≥8 GB RAM)
docker compose build
time docker compose run --rm cnpj-etl full-pipeline

# 5. Verify
psql "$STAGING_DATABASE_URL" -c "SELECT count(*), count(DISTINCT cnae_primario) FROM cnpj_enrichment WHERE uf='SP';"
# Expected: count >= 50000, distinct CNAEs = 10
```

## Risks surfaced during scaffold

1. **Upstream stubs in commit history:** scripts-upstream/*.py are STUB files in
   this commit. CI must materialize them before any real run. Mitigation: `if:
   false` guard on the GH workflow + README setup §1.
2. **Hadolint not available locally:** Dockerfile lint deferred to CI. The
   image follows best practices (non-root user, single RUN for apt, no
   pip cache, pinned base image SHA tag).
3. **DATABASE_URL must hit pooler:** per project memory `feedback_supabase-pooler.md`,
   any Vercel/Supabase ingestion must use the IPv4 Shared Pooler (aws-1,
   port 5432). `push_supabase.sh` is agnostic — the URL chosen at runtime
   is responsibility of the operator's secret.
