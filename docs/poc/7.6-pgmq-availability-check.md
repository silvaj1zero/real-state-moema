## Findings

| Extension | Available | Default Version | Installed | Notes |
|-----------|-----------|-----------------|-----------|-------|
| `pgmq`    | YES       | 1.5.1           | NO        | Native PostgreSQL message queue (Tembo/CoreDB). Story 7.6 candidate. |
| `pg_cron` | YES       | 1.6.4           | NO        | Required for AC1 in Story 7.6 (scheduled worker polling). |
| `pgcrypto`| YES       | 1.3             | YES (1.3) | Available for Story 7.10 fallback. |
| `supabase_vault` | YES | 0.3.1         | YES (0.3.1) | Primary path for Story 7.10 (see `7.10-vault-availability-check.md`). |

**Conclusion:** `pgmq` 1.5.1 is available on this Supabase project but not yet installed. It is a single `CREATE EXTENSION pgmq;` away from being usable. `pg_cron` 1.6.4 is in the same state — needed to schedule the queue worker without external cron.

## Decision

**Path A — Adopt `pgmq` natively (RECOMMENDED).**

Rationale:
1. Available on the current Supabase plan (no upgrade required).
2. Eliminates need to design/maintain custom queue table, visibility timeout reclaim, and dead-letter logic.
3. Battle-tested API: `pgmq.send`, `pgmq.read`, `pgmq.delete`, `pgmq.archive`.
4. `pg_cron` (also available) handles the worker tick on the database side, removing the need for an external scheduler in Vercel.

### Migration plan (for @dev / @devops in Story 7.6)

Pin versions explicitly. Add to migration `012_epic7_pgmq_setup.sql`:

```sql
-- Enable extensions (idempotent)
CREATE EXTENSION IF NOT EXISTS pgmq VERSION '1.5.1';
CREATE EXTENSION IF NOT EXISTS pg_cron VERSION '1.6.4';

-- Epic 7 queues
SELECT pgmq.create('epic7_lead_enrichment');
SELECT pgmq.create('epic7_portal_scrape');
SELECT pgmq.create('epic7_dead_letter');  -- DLQ for poison messages

-- Grant minimal access to service_role (RLS does not apply to pgmq tables; grants do)
GRANT USAGE ON SCHEMA pgmq TO service_role;
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA pgmq TO service_role;

-- Schedule worker tick every 60s (pg_cron)
SELECT cron.schedule(
  'epic7-lead-enrichment-tick',
  '* * * * *',
  $$ SELECT public.epic7_process_lead_enrichment_batch(10); $$
);
```

**Note:** The `public.epic7_process_lead_enrichment_batch(batch_size int)` PL/pgSQL function is the worker body — to be implemented in Story 7.6. It must call `pgmq.read('epic7_lead_enrichment', vt := 30, qty := batch_size)`, process each message idempotently, then `pgmq.delete` on success or `pgmq.archive` on poison threshold.

### Fallback (Path B) — keep documented but not selected

If `pgmq` becomes unavailable or version-pinning fails, the fallback is a `public.epic7_task_queue` table with `enqueued_at`, `visibility_timeout_until`, `attempts`, `payload jsonb`, status enum, plus a partial index on pending rows and a worker that uses `SELECT ... FOR UPDATE SKIP LOCKED` to reclaim invisible-expired rows. Skipped because Path A is cleaner.
