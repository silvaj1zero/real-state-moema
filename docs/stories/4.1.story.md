# Story 4.1: Sistema de Referrals

**Epic:** 4 — Parcerias Ganha/Ganha & Escala
**Status:** InProgress
**Priority:** Must
**Estimated Complexity:** L

## User Story

Como consultora RE/MAX,
Quero cadastrar parceiros e criar/receber referrals com perfil de cliente,
Para que eu amplie minha rede de indicações mútuas e capture leads de regiões que não são minha área de domínio.

## Acceptance Criteria

- [x] AC1: Tela "Parceiros" acessível via menu lateral → ícone pessoas (#003DA5). Lista de parceiros cadastrados em cards (nome, franquia, região, telefone mascarado, contagem referrals enviados/recebidos). Botão "+ Parceiro" (FAB azul #003DA5, 56px). Parceiros são contatos — NÃO precisam ser usuários do sistema (princípio PV: unilateral).
- [x] AC2: Formulário de cadastro de parceiro com campos: nome (obrigatório, input 48px, auto-focus), franquia (input 48px, ex: "RE/MAX Lux"), telefone (BYTEA criptografado via `pgp_sym_encrypt`, máscara BR), email (opcional), região de atuação (input texto). INSERT em `referrals` tabela auxiliar — dados do parceiro ficam inline na tabela `referrals` (colunas `parceiro_nome`, `parceiro_franquia`, `parceiro_telefone_encrypted`, `parceiro_email`, `parceiro_regiao`). Não há tabela `consultores_parceiros` separada — parceiros são normalizados por nome+franquia nos referrals.
- [x] AC3: Botão "Novo Referral" no card do parceiro ou via FAB → menu radial → "Referral". Formulário com: direção (chips: "Enviar" / "Receber" — mapeados para enum `direcao_referral` = 'enviado'/'recebido'), perfil do cliente indicado (textarea `cliente_perfil`), tipologia desejada (input `tipologia_desejada`), faixa de preço mín/máx (inputs numéricos `faixa_preco_min`/`faixa_preco_max`, formato R$ com máscara), região desejada (`regiao_desejada`), prazo de validade (date picker `prazo_validade`), notas (textarea). INSERT em `referrals` com `consultant_id = auth.uid()`, `status = 'enviada'`.
- [x] AC4: Status rastreável com pipeline visual horizontal: Enviada → Aceita → Em Andamento → Convertida → Comissão Paga. Cada status é um chip colorido (Enviada: cinza #9E9E9E, Aceita: azul #003DA5, Em Andamento: laranja #FF8C00, Convertida: verde #28A745, Comissão Paga: dourado #FFD700). Adicionalmente: Recusada (vermelho #DC3545), Expirada (cinza claro #BDBDBD). Transição manual via dropdown no card do referral. UPDATE em `referrals.status` (enum `status_referral`).
- [x] AC5: Ao mudar status para "Aceita", se direção = 'recebido', sistema sugere criar lead automaticamente: modal "Criar lead a partir deste referral?" com dados pré-preenchidos (nome do cliente, tipologia, faixa preço, região). Confirmação cria INSERT em `leads` com `referral_id = referrals.id`, `origem = 'indicacao'`, `etapa_funil = 'contato'`. Referral atualizado com `lead_id` FK. Se direção = 'enviado', apenas muda status sem criar lead.
- [x] AC6: Histórico completo de referrals na tela "Parceiros": filtros por status (multi-select chips), direção (Enviados/Recebidos/Todos), período (date range). Listagem em cards com: parceiro, cliente perfil (truncado 80 chars), status chip, data criação, lead vinculado (link clicável se existir). Ordenação por `created_at DESC`. Paginação cursor-based (20 por página).
- [x] AC7: Métricas de reciprocidade no topo da tela "Parceiros": card resumo com "Enviados: N | Recebidos: N | Convertidos: N | Taxa conversão: X%". Por parceiro: badge de reciprocidade (verde se diferença enviados/recebidos <= 2, amarelo se 3-5, vermelho se >5). Query agregada em `referrals` GROUP BY `parceiro_nome, parceiro_franquia`.
- [x] AC8: Expiração automática: referrals com `prazo_validade < now()` e status IN ('enviada', 'aceita') são marcados como 'expirada' via Supabase Edge Function (cron diário) ou check client-side ao carregar. Badge visual "Expirado" com ícone relógio. Referrals expirados não contam nas métricas ativas.
- [x] AC9: Ao salvar novo referral, toast "Referral para [Parceiro] criado!" com haptic feedback. Push notification configurável (respeita `consultant_settings.notificacoes_push`). Dados sensíveis (telefone parceiro) criptografados via pgcrypto — LGPD compliance.

## Technical Notes

- **Database:** Tabela `referrals` (migration 004). Colunas: id, consultant_id, direcao (enum `direcao_referral`), parceiro_nome, parceiro_franquia, parceiro_telefone_encrypted (BYTEA), parceiro_email, parceiro_regiao, cliente_perfil, tipologia_desejada, faixa_preco_min, faixa_preco_max, regiao_desejada, prazo_validade, status (enum `status_referral`), lead_id (FK nullable), notas, created_at, updated_at. Índices: `idx_referrals_consultant`, `idx_referrals_status` (partial WHERE status NOT IN expirada/recusada).
- **Enums:** `status_referral` (enviada, aceita, recusada, em_andamento, convertida, comissao_paga, expirada), `direcao_referral` (enviado, recebido).
- **API:** Supabase client SDK. TanStack Query: mutation para INSERT/UPDATE, query key `['referrals', consultant_id]`. Optimistic update nos status transitions.
- **Components:** PartnerList, PartnerCard, ReferralForm, ReferralPipeline (status chips), ReciprocityMetrics, ReferralHistory (filtros + paginação).
- **Criptografia:** `pgp_sym_encrypt(telefone, current_setting('app.encryption_key'))` para telefone do parceiro. Decrypt apenas em leitura.
- **Princípio PV:** Sistema UNILATERAL — parceiros são contatos gerenciados pela Luciana, não usuários com login. Sem necessidade de autenticação do parceiro.

## Notes

- **Princípio PV — Unilateral:** Parceiros são contatos manuais, não usuários do sistema. A Luciana gerencia tudo. Isso simplifica drasticamente a arquitetura (sem auth de parceiro, sem notificação bidirecional, sem sync).
- **Princípio PV — Dados inline:** Dados do parceiro ficam na própria tabela `referrals` (não em tabela separada) porque parceiros não são entidades com lifecycle próprio — são atributos do referral.
- **LGPD (NFR-003):** Telefone do parceiro criptografado via pgcrypto. Email em texto puro (considerado não-sensível neste contexto).
- Expiração via cron é preferível a check client-side para consistência de dados, mas fallback client-side garante UX correta mesmo sem Edge Function.
