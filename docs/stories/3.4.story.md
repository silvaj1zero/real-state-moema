# Story 3.4: Agente de Varredura de Portais FISBO

**Epic:** 3 — Inteligência, ACM & Agentes de Automação
**Status:** Done
**Priority:** Must
**Estimated Complexity:** XL

## User Story

Como consultora RE/MAX,
Quero que o sistema varra automaticamente portais imobiliários e detecte anúncios de proprietários diretos (FISBO) na minha micro-região,
Para que eu receba leads qualificados sem precisar monitorar manualmente ZAP, OLX e VivaReal todos os dias.

## Acceptance Criteria

- [x] AC1: Agente de varredura varre portal ZAP Imóveis (Must — portal principal). Apify Actor dedicado para ZAP: busca anúncios filtrados por localização (Moema, Vila Olímpia, Itaim Bibi), tipo "Venda", e identifica anunciante = "Proprietário" (campo do ZAP que diferencia proprietário de corretor/imobiliária). Cada anúncio coletado gera INSERT em `scraped_listings` com: `portal = 'zap'`, `external_id` (ID único do anúncio no ZAP), `url` (link direto), `tipo_anunciante` (proprietario/corretor/imobiliaria/desconhecido), `endereco`, `preco`, `area_m2`, `tipologia`, `quartos`, `descricao`, `is_fisbo = true` se tipo_anunciante = 'proprietario'. UNIQUE constraint `(portal, external_id)` previne duplicatas.
- [x] AC2: Varredura de OLX e VivaReal (Should). Apify Actors adicionais para OLX Imóveis e VivaReal. Mesma lógica de coleta e INSERT em `scraped_listings` com `portal = 'olx'` ou `portal = 'vivareal'`. OLX: filtro "Particular" no tipo de anunciante. VivaReal: detecção heurística (ausência de logo imobiliária, texto "proprietário" na descrição). Prioridade de implementação: ZAP primeiro (Must), OLX e VivaReal depois (Should).
- [x] AC3: Match de anúncios com edifícios via PostGIS `ST_DWithin(50m)` + geocoding. Para cada `scraped_listing` com `geocoding_status = 'pending'`: (a) Se `coordinates` já preenchido (portal forneceu lat/lng), executa `fn_match_listing_edificio(listing_id)` — busca edifício mais próximo em 50m, atualiza `matched_edificio_id` e `match_method = 'postgis_50m'`. (b) Se `coordinates` é NULL e `endereco` existe, executa Sub-task 3.4d (geocoding) para obter coordenadas, depois match. (c) Se match encontrado, `match_distance_m` registra distância em metros. (d) Se nenhum edifício em 50m, `match_method = 'unmatched'` — listing fica disponível para match manual futuro.
- [x] AC4: **Sub-task 3.4d — Geocoding via Mapbox Geocoding API.** Para listings com `endereco` mas sem `coordinates`: chamada a `https://api.mapbox.com/geocoding/v5/mapbox.places/{endereco_encoded}.json?country=br&bbox=-46.68,-23.62,-46.63,-23.57&access_token={MAPBOX_TOKEN}`. Bbox restrito a Moema e arredores. Resposta: `features[0].center` = [lng, lat]. UPDATE `scraped_listings` SET `coordinates = ST_SetSRID(ST_MakePoint(lng, lat), 4326)::geography`, `endereco_normalizado = features[0].place_name`, `geocoding_status = 'success'`. Se API retorna 0 resultados: `geocoding_status = 'failed'`. Rate limit Mapbox: 600 req/min (free tier) — batch com delay 100ms entre requests. Mapbox token armazenado em Supabase secrets.
- [x] AC5: Cron job para varredura periódica. Supabase Edge Function `scrape-portals` acionada via `pg_cron`: (a) Diário às 06:00 BRT para ZAP (Must), (b) Semanal (segunda-feira 06:00) para OLX e VivaReal (Should). Edge Function: chama Apify Actor via API REST (`POST https://api.apify.com/v2/acts/{actorId}/runs`), aguarda conclusão (`GET .../runs/{runId}`), baixa resultados (`GET .../runs/{runId}/dataset/items`), processa e insere em `scraped_listings`. Logging de cada execução: `INSERT INTO intelligence_feed` tipo `sync_completo` com metadata `{portal, total_coletados, novos, atualizados, fisbo_detectados}`.
- [x] AC6: Detecção de remoção de anúncios. Na varredura, se anúncio previamente ativo (`is_active = true`) não aparece mais nos resultados: UPDATE `scraped_listings` SET `is_active = false`, `removed_at = now()`. Anúncio removido NÃO é deletado — permanece para histórico e análise de tempo de mercado (`removed_at - first_seen_at`).
- [x] AC7: Detecção de mudança de preço. Se anúncio existe e novo preço difere: trigger `fn_notify_price_change` detecta redução >10% e gera evento no `intelligence_feed` tipo `reducao_preco`. UPDATE `scraped_listings` SET `preco = novo_preco`, `preco_anterior = preco_atual`, `preco_changed_at = now()`. Histórico de preço preservado para análise de tendência (alimenta ACM Story 3.1).
- [x] AC8: Fallback CSV para importação manual. Se Apify falhar (quota excedida, Actor indisponível, portal bloqueia scraping): botão "Importar CSV" na tela de administração. Upload de CSV com colunas mínimas: endereco, preco, area_m2, quartos, tipo_anunciante, url. Parser CSV processa e insere em `scraped_listings` com `portal = 'outro'` e `geocoding_status = 'pending'`. Validação: campos obrigatórios presentes, preco numérico, area_m2 numérico. Toast "N anúncios importados!" ou "Erro: [detalhes]".
- [x] AC9: Dashboard de monitoramento do agente. Tela "Agentes" (acessível via menu settings) exibe: (a) status de cada cron (último run, próximo run, resultado), (b) métricas: total listings ativos, total FISBO, novos esta semana, removidos esta semana, (c) botão "Executar Agora" para forçar varredura manual, (d) log de erros das últimas 10 execuções. Dados de `scraped_listings` aggregate + `intelligence_feed` tipo `sync_completo`.

## Technical Notes

- **Database:** INSERT/UPDATE em `scraped_listings` (id, portal, external_id, url, tipo_anunciante, endereco, endereco_normalizado, coordinates, geocoding_status, bairro, preco, area_m2, preco_m2, tipologia, quartos, descricao, is_fisbo, first_seen_at, last_seen_at, removed_at, preco_anterior, preco_changed_at, matched_edificio_id, match_method, match_distance_m, is_active). Enums: `portal_scraping`, `tipo_anunciante`, `geocoding_status`, `match_method`. Function `fn_match_listing_edificio`. Triggers: `fn_notify_new_fisbo`, `fn_notify_price_change`. Indexes: GIST em coordinates, `(portal, external_id)` unique, `is_fisbo WHERE true AND is_active`, `is_active WHERE true`.
- **Apify:** Actors para scraping. API REST: `POST /v2/acts/{actorId}/runs` com input JSON (localização, filtros). `GET /v2/acts/{actorId}/runs/{runId}` para status. `GET /v2/actor-runs/{runId}/dataset/items` para resultados. Token em Supabase secrets `APIFY_TOKEN`.
- **Mapbox Geocoding:** `GET /geocoding/v5/mapbox.places/{query}.json?country=br&bbox={moema_bbox}&access_token={token}`. Token em Supabase secrets `MAPBOX_TOKEN`. Bbox Moema: `-46.68,-23.62,-46.63,-23.57`.
- **Cron:** `pg_cron` extension no Supabase para scheduling. Alternativa: Supabase Edge Function com invocação via external cron (GitHub Actions, Vercel Cron).
- **Components:** AgentDashboard (tela monitoramento), AgentStatusCard (status de cada cron), CsvImportModal (upload CSV fallback), ScrapedListingsTable (tabela de listings), ListingMatchStatus (indicador de match).

## Notes

- **Princípio PV — Scraping é acelerador, não dependência:** Todo o sistema (ACM, Feed, Dossiê) funciona sem scraping. O agente de varredura enriquece os dados mas nunca é pré-requisito.
- **Princípio PV — Graceful degradation:** Se Apify falha, CSV fallback mantém fluxo. Se geocoding falha, listing fica pendente para retry. Se portal bloqueia, sistema opera com dados existentes.
- ZAP Imóveis é o portal prioritário — maior volume de anúncios FISBO na região. OLX e VivaReal são complementares.
- Geocoding via Mapbox com bbox restrito a Moema minimiza falsos positivos e custo de API.
- Match PostGIS 50m é conservador — edifícios muito próximos (geminados) podem causar match incorreto. Revisão manual disponível via dashboard.
- Trigger `fn_notify_new_fisbo` gera evento automático no feed (Story 3.7) quando novo FISBO detectado.
- Trigger `fn_notify_price_change` gera evento quando redução >10% — sinal de proprietário motivado.
