# Story 4.7: Safari/Open House

**Epic:** 4 — Parcerias Ganha/Ganha & Escala
**Status:** Done
**Priority:** Must
**Estimated Complexity:** M

## User Story

Como consultora RE/MAX,
Quero criar e gerenciar eventos Safari/Open House vinculados a imóveis exclusivos,
Para que eu convide parceiros e clientes, rastreie confirmações, execute o evento com checklist e registre resultados que alimentem o Dossiê.

## Acceptance Criteria

- [x] AC1: Botão "Criar Safari" acessível via: (a) card do lead em estado Exclusividade → seção "Eventos", (b) tela Plano de Marketing → item Safari (Story 4.8), (c) FAB long-press → "Safari". Formulário com: título (input 48px, auto-focus, obrigatório), descrição (textarea), data/hora (datetime picker — `safari_events.data_hora` TIMESTAMPTZ), endereço (pré-preenchido do edifício se `edificio_id` disponível), vagas (input numérico, default 10 — `safari_events.vagas`). INSERT em `safari_events` com `consultant_id = auth.uid()`, `lead_id` FK, `edificio_id` FK (nullable), `status = 'planejado'`.
- [x] AC2: Vinculação obrigatória a imóvel exclusivo: campo `lead_id` obrigatório (select com leads em etapa 'exclusividade' ou 'venda'). Campo `edificio_id` auto-preenchido a partir do lead selecionado (`leads.edificio_id`). Se lead não tem edifício vinculado, campo endereço livre (textarea). Validação: não permitir Safari sem lead vinculado.
- [x] AC3: Convite a parceiros: seção "Convidados" no formulário do Safari. Botão "+ Convidado" abre modal com: nome (obrigatório), franquia (opcional), telefone (opcional). INSERT em `safari_event_rsvps` com `safari_event_id` FK, `status = 'convidado'`. Lista de parceiros cadastrados (Story 4.1) aparece como sugestão — ao selecionar parceiro existente, pré-preenche nome/franquia/telefone. Possibilidade de adicionar convidados que não são parceiros cadastrados.
- [x] AC4: Link compartilhável: botão "Copiar Link" gera URL única para o evento (ex: `app.domain/safari/{event_id}`). Landing page pública (sem auth) com: título, descrição, data/hora, endereço, vagas restantes, botão "Confirmar Presença" (nome + telefone). Ao confirmar, INSERT em `safari_event_rsvps` com `status = 'confirmado'`. Se vagas esgotadas, exibe "Evento lotado" e botão "Entrar na fila de espera".
- [x] AC5: RSVP tracking: na tela do Safari, lista de convidados com status visual (chips): Convidado (cinza #9E9E9E), Confirmado (verde #28A745), Recusado (vermelho #DC3545), Pendente (amarelo #FFC107). Contagem no topo: "N confirmados de M convidados | K vagas restantes". Status atualizável manualmente via tap no chip → dropdown. Enum `status_rsvp` (convidado, confirmado, recusado, pendente).
- [x] AC6: Checklist de preparação: seção expansível "Preparação" no card do Safari. Itens fixos: "Fotos atualizadas", "Imóvel limpo e arrumado", "Material impresso (fichas)", "Placa de Safari na entrada", "Água/café disponível", "Chaves confirmadas com proprietário". Cada item é checkbox com data de conclusão. Armazenamento em campo JSON `safari_events.checklist_preparacao` (JSONB a adicionar) ou em tabela `checklists_preparacao` existente (Epic 2) com `entity_type = 'safari'`.
- [x] AC7: Pós-evento: ao mudar status para 'realizado', formulário de conclusão obrigatório: feedback geral (textarea — `safari_events.feedback`), número de propostas recebidas (input numérico — `safari_events.propostas_recebidas`), lista de interessados (link para criar lead se não existe). UPDATE em `safari_events.status = 'realizado'`. Se `propostas_recebidas > 0`, badge dourado "Propostas!" no card.
- [x] AC8: Histórico de Safaris: tela "Safaris" acessível via menu lateral → ícone casa com pessoas. Lista ordenada por `data_hora DESC` com cards: título, data, imóvel (endereço), status chip, convidados (N confirmados), propostas (N). Filtros: status (planejado/confirmado/realizado/cancelado), período. Tap no card abre detalhe completo.
- [x] AC9: Integração Dossiê (Story 3.2): Safari vinculado ao lead aparece no Dossiê na seção "Eventos". Exibe: data, título, convidados confirmados, feedback, propostas. Join via `safari_events.lead_id = dossies.lead_id`. Dado visível como evidência de trabalho ativo para o proprietário.

## Technical Notes

- **Database:** Tabela `safari_events` (migration 004). Colunas: id, consultant_id (FK), lead_id (FK), edificio_id (FK nullable), titulo, descricao, data_hora (TIMESTAMPTZ), endereco, vagas (default 10), status (enum `status_safari`: planejado/confirmado/realizado/cancelado), feedback, propostas_recebidas (default 0), created_at, updated_at. Tabela `safari_event_rsvps`: id, safari_event_id (FK), nome_convidado, franquia, telefone, status (enum `status_rsvp`: convidado/confirmado/recusado/pendente), notas, created_at. Índices: `idx_safari_events_consultant`, `idx_safari_rsvps_event`.
- **Landing page:** Rota pública `/safari/[id]` com SSR ou CSR sem auth. Supabase anon key para INSERT em rsvps (RLS policy: INSERT permitido para anon com validação de vagas). SELECT limitado a dados públicos do evento (sem dados sensíveis).
- **API:** Supabase client SDK. TanStack Query: `['safari_events', consultantId]`, `['safari_rsvps', eventId]`. Mutations com optimistic update.
- **Components:** SafariForm, SafariCard, RSVPList, RSVPChip, ShareLinkButton, PreparationChecklist, PostEventForm, SafariHistory.

- **Princípio PV — Safari é ferramenta de posicionamento:** Open Houses/Safaris são parte da metodologia RE/MAX para "Ser Conhecida" (Fase 2 do posicionamento). Não é apenas evento — é estratégia de marketing territorial.
