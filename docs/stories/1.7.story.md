# Story 1.7: Seed Data — Pré-carga de Edifícios (VETO PV #1)

**Epic:** 1 — Foundation, Mapa & Registro de Campo
**Status:** Ready
**Priority:** Must
**Estimated Complexity:** M

## User Story

Como consultora RE/MAX,
Quero que o mapa já mostre edifícios ao abrir pela primeira vez,
Para que eu possa focar em qualificar e prospectar, não em cadastrar do zero.

## Acceptance Criteria

- [ ] AC1: Ao configurar epicentro (primeira vez ou ao mover), sistema executa Supabase Edge Function `seed-data` com estratégia de 3 camadas verificadas (ver `docs/research/api-integration-research.md` seção 11):
  - **Camada 1 (Must):** OSM Overpass API (`building=apartments` / `building=residential`) — fonte primária, 16.595 edifícios verificados no raio 2km de Moema (ver `docs/research/api-verification-results.md`)
  - **Camada 2 (Should):** Google Places API com `type=apartment_building` (NÃO `premise` — tipo inválido, ver `docs/research/google-places-verification.md`) — enriquecimento com nomes (80%) e endereços formatados (95%), ~100-400 edifícios
  - **Camada 3 (Epic 3 — Story 3.5):** GeoSampa IPTU (nº unidades, área m², padrão, ano construção) — ver `docs/research/geosampa-iptu-verification.md`
  INSERT em `edificios` com `origem = 'seed'`, `seed_source = 'osm_overpass'` ou `'google_places'`, `verificado = false`.
- [ ] AC2: Edifícios pré-carregados entram com status "Não Visitado" (cinza) — sem registro em `edificios_qualificacoes` (nenhuma qualificação até a consultora interagir). Dados básicos: nome (do Google Places/OSM), endereço, coordenadas (geography Point 4326), bairro, cep, cidade, estado.
- [ ] AC3: Luciana pode interagir com cada edifício seed: confirmar (cria `edificios_qualificacoes` com status_varredura='mapeado', marca `edificios.verificado = true`), editar (alterar nome, endereço, tipologia, padrão — marca verificado=true), ou descartar (soft delete ou flag `descartado`). Ações disponíveis no card do edifício (Story 1.4).
- [ ] AC4: Badge visual "auto" nos pins de seed data (borda pontilhada + ícone "A" conforme frontend-spec seção 3.1 — `isAutoSeed = true`). Badge muda para "verificado" (borda sólida + checkmark, `isVerified = true`) após confirmar ou editar. Diferenciação clara no mapa entre dados seed e dados verificados.
- [ ] AC5: Seed roda na configuração inicial do epicentro + re-executável sob demanda via botão em Configurações ("Recarregar edifícios da região"). Deduplica por proximidade: se já existe edifício dentro de 30m (PostGIS `ST_DWithin`), não insere duplicata. Feedback: toast "N novos edifícios encontrados" ou "Nenhum novo edifício encontrado".
- [ ] AC6: Edge Function `seed-data` implementa: (1) recebe coordenadas do epicentro e raio, (2) consulta Google Places API (ou OSM Overpass como fallback), (3) normaliza dados para schema `edificios`, (4) deduplica via PostGIS ST_DWithin(30m), (5) insere em batch. Rate limiting: máx 1 execução por epicentro por hora.
- [ ] AC7: Fallback gracioso — se APIs externas estiverem indisponíveis (rate limit, erro de rede), o sistema funciona normalmente sem seed data. A consultora cadastra manualmente. Log de erro, sem bloqueio. Seed pode ser re-tentado depois.

## Technical Notes

- **Database:** INSERT batch em `edificios` com origem='seed', seed_source='google_places'|'osm_overpass', verificado=false, created_by=null (seed não tem autor). Deduplica via: `NOT EXISTS (SELECT 1 FROM edificios e2 WHERE ST_DWithin(e2.coordinates, :new_coords, 30))`. Soft delete para descarte: campo `descartado` (bool) ou DELETE se preferir.
- **API:** Supabase Edge Function `seed-data` (Deno).
  - **Primário:** OSM Overpass API: `[out:json][timeout:60];(way["building"](around:2000,-23.5988,-46.6658);relation["building"](around:2000,-23.5988,-46.6658););out center;` — retorna ~16.595 edifícios (verificado).
  - **Enriquecimento:** Google Places Nearby Search API: `https://maps.googleapis.com/maps/api/place/nearbysearch/json?location={lat},{lng}&radius=2000&type=apartment_building` (NÃO usar `type=premise` — inválido). Requer grid search (max 60 resultados por centro).
  - Environment variables: `GOOGLE_PLACES_API_KEY`, nenhuma key necessária para OSM Overpass.
  - **Referências de verificação:** `docs/research/api-verification-results.md`, `docs/research/google-places-verification.md`
- **UI Components:** MapPin com badge "auto" (seção 3.1 — isAutoSeed, borda pontilhada), ConfirmSeedDialog, SettingsPage com botão "Recarregar edifícios", Toast de resultado.
- **PostGIS:** `ST_DWithin` para deduplica (30m). `ST_MakePoint(lng, lat)::geography` para converter coordenadas das APIs. Insert batch com coordenadas geography.
- **Offline:** Seed data requer conexão (APIs externas). Após seed, edifícios são cacheados no IndexedDB (Story 1.6) e ficam disponíveis offline.

## Notes

- **VETO PV #1 — "Dados antes de ação":** Este é o princípio mais importante de Pedro Valério aplicado ao Epic 1. O mapa DEVE estar populado no dia 1. Uma consultora abrindo um mapa vazio não sabe por onde começar. Com edifícios seed, ela sabe exatamente o que existe no raio e foca em qualificar, não em descobrir.
- **Princípio PV — Impossibilitar o caminho errado:** O mapa populado desde o início elimina a barreira de "tela vazia" que desmotiva o uso. A consultora vê valor imediato ao abrir o app.
- **Princípio PV — Funciona sozinho antes de depender de externo:** Se Google Places / OSM falharem, o sistema funciona 100% — a consultora cadastra manualmente. Seed é acelerador, não dependência.
- Google Places API tem custo ($32 per 1000 requests para Nearby Search). Para o MVP single-user, o custo é desprezível (~5-10 requests total). Monitorar se escalar.
- **OSM Overpass é a FONTE PRIMÁRIA** (verificado: 16.595 edifícios no raio 2km de Moema). Google Places é COMPLEMENTO (~100-400 edifícios, mas com dados mais ricos: nomes, endereços formatados). Estratégia: OSM pra volume, Google Places pra enriquecimento.
- **GeoSampa IPTU (Epic 3, Story 3.5):** Fonte premium futura com nº unidades, área construída, padrão construtivo e ano de construção. Ver `docs/research/geosampa-iptu-verification.md` para detalhes e passos de implementação.
- A deduplica por 30m é heurística — edifícios muito próximos podem ser erroneamente considerados duplicatas. Permitir revisão manual.
- **Edge Function vs API Route:** Usar Edge Function do Supabase (Deno) porque precisa de acesso direto ao banco (deduplica com PostGIS) e é desacoplada do frontend. API Route do Next.js seria alternativa viável.
