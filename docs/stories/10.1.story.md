# Story 10.1: Call list FISBO priorizada + status de tentativa de contato

**Epic:** 10 — Agenda de Prospecção FISBO
**Camada:** Dados + UI
**Status:** InReview
**Priority:** Must
**Estimated Complexity:** M (5-8 pts)
**Executor sugerido:** @dev (+ @data-engineer p/ a coluna de status)

> **Reuso (Art. IV):** `is_fisbo` (classify-advertiser/publisherType), contato enriquecido (`telefone_anunciante`/`whatsapp_anunciante` + links `tel:`/`wa.me` do `ContactDataCard`), `etapa_funil` e o funil já existentes. Esta story **não cria CRM novo** — só adiciona a lista acionável + o status de tentativa.

## User Story

Como Luciana (consultora),
Quero uma **lista priorizada dos FISBO** com telefone/WhatsApp clicável e um **registro do resultado de cada tentativa** (atendeu / não atendeu / retornar / agendado),
Para trabalhar as ligações em sequência sem perder quem já liguei nem quem preciso retornar.

## Contexto
Hoje os FISBO existem em `scraped_listings` (`is_fisbo=true`, contato enriquecido) e viram `leads` (`origem='fisbo_scraping'`), mas não há uma **lista de chamadas** que os ordene por prioridade nem onde registrar o **resultado da tentativa** — só há `etapa_funil` (grosso). Falta a granularidade operacional do dia de ligações.

## Acceptance Criteria
- [x] AC1: **Tela/seção "Call list FISBO"** lista os FISBO acionáveis (de `scraped_listings`/`leads` com `is_fisbo=true` do consultor logado, RLS), cada item com: endereço, bairro, R$/m² ou preço, nome do anunciante e **telefone/WhatsApp clicável** (reusar `ContactDataCard` ou seus links `tel:`/`wa.me`). → `FisboCallList.tsx` + util `contact-links.ts` extraído de `ContactDataCard`.
- [x] AC2: **Status de tentativa de contato** por item — enum `nao_contatado | atendeu | nao_atendeu | retornar | agendado | descartado` — **persistido reusando tabela existente**. **Decisão @data-engineer:** mora em `leads` (não em `scraped_listings`, que é SELECT-only por RLS para `authenticated`). Migration 022: colunas `contato_status`/`contato_status_at`/`contato_notas` + `scraped_listing_id` (vínculo determinístico). Sem CRM/tabela nova.
- [x] AC3: **Registrar tentativa** com 1 toque (botões de status) + `notas` opcional; atualização otimista (`onMutate`), reflete na lista. Captação automática (materializa lead) se o FISBO ainda não for lead.
- [x] AC4: **Priorização** padrão documentada em `callListOrder.ts`: status de ação primeiro (`nao_contatado`→`retornar`→…→`descartado`); empate por proximidade (se `origin`+coords) ou recência (`last_seen_at`). (Roteiro completo = 10.2.)
- [x] AC5: **Ponte com o funil/agenda:** ao marcar `agendado`, move `etapa_funil`→`v1_agendada` (só a partir de `contato`) e abre o `ScheduleModal` (reusa `useAgendamentos`/store). `useCallListBridge.ts`.
- [x] AC6: **Degradação graciosa:** FISBO sem telefone/WhatsApp aparece "Sem contato" + ação "Enriquecer" (reusa `useEnrichContact`, Story 6.4) — não some da lista.
- [x] AC7: **Filtros mínimos:** por bairro e por `contato_status` (selects no header).
- [x] AC8: **Testes:** `callListOrder.test.ts` (14 casos: sem coords, sem contato, mix de status, proximidade vs recência) + smoke `FisboCallList.test.tsx` (5 casos). 19/19 verdes.

## Dev Notes
- **Decisão @data-engineer (AC2):** status em `leads`, não em `scraped_listings` — a RLS de `scraped_listings` é SELECT-only para `authenticated` (migration `20260414000001`, seção 5.1); o client não pode dar UPDATE. `leads` tem CRUD por `consultant_id` e é a entidade acionável do funil. Coluna `scraped_listing_id` dá o vínculo determinístico (sobreposição de status + dedup via índice único `(consultant_id, scraped_listing_id)`).
- **Pendência de deploy:** a migration 022 precisa ser aplicada no Supabase (SQL Editor) antes da tela funcionar em runtime — padrão deste projeto. Sem ela, o SELECT de `contato_status` falha.
- **Proximidade (AC4):** `orderCallList` aceita `origin` opcional; em 10.1 não há origin (ordena por status→recência). A 10.2 fornecerá o `origin` (epicentro) para o roteiro.
