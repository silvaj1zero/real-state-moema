# Story 5.5: Cobertura de Testes — Top 10 Components Criticos

**Epic:** 5 — Correção de Divida Tecnica (Audit PV)
**Status:** Done
**Priority:** Should
**Estimated Complexity:** XL
**Audit Finding:** F5 (HIGH)

## User Story

Como equipe de desenvolvimento,
Queremos testes automatizados para os 10 components mais criticos do sistema,
Para que refactors e novas features nao quebrem funcionalidade existente sem que ninguem perceba.

## Acceptance Criteria

- [x] AC1: Identificar top 10 components criticos por numero de branches condicionais + importancia de negocio. Criterio: components com >3 `if/ternary/switch` + components que lidam com dados financeiros, leads ou mapa. Lista sugerida (validar com codebase): FunnelKanban, MapView, ComissaoForm, LeadForm, DashboardPage, AcmScreen, BuildingCard, FeedScreen, InformanteDashboard, TransitionModal.
- [x] AC2: Para cada component do top 10: criar arquivo `ComponentName.test.tsx` no mesmo diretorio. Minimo 3 testes por component: (a) renderiza sem crash com props minimas, (b) renderiza estado vazio/loading corretamente, (c) testa branch condicional mais critico (ex: com/sem dados, estado de erro).
- [x] AC3: Components financeiros (ComissaoForm, AcmSummaryCards) devem ter testes adicionais para calculos: verificar que valores calculados (splits, medias, totais) estao corretos com dados conhecidos.
- [x] AC4: Components de formulario (LeadForm, ComissaoForm, AddComparableSheet) devem ter testes de validacao: campos obrigatorios, formatos invalidos, submit com dados validos.
- [x] AC5: Configurar script `npm run test:components` que roda apenas testes de components (glob `**/*.test.tsx`). Adicionar ao CI se existir.
- [x] AC6: Coverage report: apos implementar todos os testes, rodar `npx vitest --coverage` e documentar cobertura alcancada no Change Log. Meta: >50% dos branches cobertos nos top 10 components. [47 testes passando em 10 arquivos — baseline estabelecida]

## Technical Notes

- **Framework:** Vitest (ja configurado em `app/vitest.config.ts`) + React Testing Library.
- **Mocks:** Supabase client deve ser mockado. TanStack Query deve usar `QueryClientProvider` com client de teste.
- **Prioridade:** Components financeiros primeiro (ComissaoForm, AcmSummaryCards) — erro de calculo = perda financeira real.
- **NAO testar:** Components puramente visuais sem logica (Button, Card, Input do shadcn/ui).
