# Story 8.6: Gerar pacote completo ACM em 1 clique (resumo + laudo + deck + didático)

**Epic:** 8 — Geração de Entregáveis ACM
**Camada:** Renderização (orquestração de UI)
**Status:** Done
**Priority:** Should
**Estimated Complexity:** S (2-3 pts)

> **Origem (PM, Morgan, 2026-06-16):** fatia de UX **sobrevivente** da 8.5. A ADR-EPIC8-001 descopou a *orquestração cross-repo app↔engine*, mas **não** a UX de "fábrica de ACM em 1 clique" (Roadmap Passo 3.1). Como render virou TS nativo (8.3a/8.3b/8.4), a fatia ficou barata: um `computeLaudo` único → emite os 4 PDFs. **Não reabre** engine externo (AC1/AC4/AC5 da 8.5 seguem no Epic 7). Esta é uma story NOVA e pequena, não a L morta.
> **PO Validation (2026-06-16, Pax):** GO (9/10). Escopo nítido, deps 100% satisfeitas (8.3a/8.3b/8.4 Done), valor de UX claro (elimina 4× redigitação do formulário). Status → Ready.

## User Story

Como Luciana (consultora),
Quero gerar o **pacote ACM completo** (resumo + laudo + deck + didático) a partir de **um único formulário, em um clique**,
Para ir da análise ao pacote de entrega sem redigitar os mesmos dados 4 vezes nem disparar 4 ações separadas.

## Contexto

Operacionaliza a UX do Passo 3.1 do Roadmap ("um comando que encadeia → gerar laudo + resumo + deck + material") **dentro do app**, sem engine externo. Reusa integralmente o pipeline nativo já entregue: `computeLaudo` (8.2) → `buildResumoModel`/`buildLaudoModel`/`buildDeckModel`/`buildDidaticoModel` (8.3a/8.3b/8.4) → `pdf().toBlob()`. O único código novo é a **orquestração**: rodar o cálculo uma vez, montar os 4 modelos e baixar os 4 PDFs.

## Acceptance Criteria

- [x] AC1: Novo item no `AcmExportMenu`: **"Gerar pacote completo (PDF)"** (destacado no topo) que abre **um único formulário** (superset: endereço/bairro/proprietário, áreas, programa, preços, meta, fatores, residual pré-preenchido) — `PacoteExportSheet`.
- [x] AC2: **Um `computeLaudo` único** alimenta os 4 entregáveis (`buildAcmPackage`) — consistência total (mesma `AcmLaudoComputation`; zero recálculo divergente). Testado.
- [x] AC3: Os **4 PDFs** baixam do mesmo clique (downloads sequenciais com micro-delay — sem nova dependência de ZIP), nomes `acm-resumo-…`/`acm-laudo-…`/`acm-deck-…`/`acm-didatico-…`.
- [x] AC4: Progresso por entregável ("Gerando Resumo Executivo… Laudo Técnico… Deck Comercial… Material Didático…") + tolerância a falha (um que falhe não impede os demais; relata os que falharam).
- [x] AC5: Reuso total dos builders/documentos da 8.3a/8.3b/8.4 e do `staticMap.ts` — o orquestrador (`acmPackage.tsx`) só compõe; zero duplicação de render/branding.
- [x] AC6: Smoke: `buildAcmPackage` num único `computeLaudo` → 4 docs `%PDF-` (Honduras + n=2/NULL). 3 testes novos.

## Dependencies

## Out of Scope

- ZIP/empacotamento (sem nova dependência — downloads sequenciais bastam; ZIP pode ser fast-follow).
- Anexar ao dossiê (era AC3 da 8.5; pode ser fast-follow separado).
- Orquestração de engine externo / pipeline ITBI (descopado pela ADR-EPIC8-001; segue no Epic 7).

## Technical Notes

- **Reuso, não reescrita:** o orquestrador chama os 4 builders já testados; o risco é só de integração (sequência de `toBlob` + downloads).
- Downloads múltiplos programáticos: encadear com `await`/micro-delay entre cada `a.click()` para o browser não descartar.
- O formulário é o mesmo superset do `EntregavelExportSheet`/`LaudoExportSheet` — extrair um helper de montagem do `input` evita drift.
