# Story 3.5: Pré-Mapeamento Geográfico Avançado

**Epic:** 3 — Inteligência, ACM & Agentes de Automação
**Status:** Ready
**Priority:** Should
**Estimated Complexity:** XL

## User Story

Como consultora RE/MAX,
Quero que o sistema complemente automaticamente os dados de edifícios com informações ricas de fontes públicas ao desbloquear um novo raio,
Para que eu tenha dados de número de unidades, padrão construtivo e ano de construção sem precisar pesquisar manualmente cada edifício.

## Acceptance Criteria

- [ ] AC1: Pré-mapeamento avançado dispara automaticamente ao desbloquear novo raio (Story 1.5). Quando `epicentros.raio_ativo_m` aumenta (ex: 500m → 1000m), Supabase Edge Function `seed-advanced-data` é acionada via database webhook (trigger on UPDATE de `epicentros` WHERE `raio_ativo_m` changed). Function busca edifícios no novo anel (raio anterior → raio atual) que não possuem dados avançados (`edificios.seed_source_secondary IS NULL`). Processo assíncrono — não bloqueia UI. Evento `intelligence_feed` tipo `seed_completo` ao finalizar com metadata `{fonte, edificios_enriquecidos, novos_edificios}`.
- [ ] AC2: Fonte Google Places API (type=apartment_building). Edge Function `seed-google-places-advanced` busca edifícios residenciais via `https://maps.googleapis.com/maps/api/place/nearbysearch/json?location={lat},{lng}&radius={raio_m}&type=apartment_building&key={GOOGLE_API_KEY}`. Tipo `apartment_building` (NOT `premise` — premise retorna comércios e casas misturados). Dados coletados: `name`, `formatted_address`, `geometry.location`, `rating`, `user_ratings_total`. Cobertura estimada: 100-400 edifícios em Moema por raio 2km. Match com edifícios existentes via `ST_DWithin(30m)` — UPDATE se match, INSERT se novo. Campo `edificios.seed_source_secondary = 'google_places'`.
- [ ] AC3: Fonte OSM Overpass API (complemento). Edge Function `seed-osm-advanced` busca edifícios com tags avançadas via Overpass: `[out:json];(way["building"="apartments"]["addr:street"](around:{raio},{lat},{lng}););out center tags;`. Dados coletados: `addr:street`, `addr:housenumber`, `building:levels`, `name`. OSM para Moema: 16.595 buildings verificados (raio 2km). Match com edifícios existentes via `ST_DWithin(20m)`. Dados de `building:levels` mapeados para `edificios.num_pavimentos` (campo Story 1.7 ou novo).
- [ ] AC4: Fonte GeoSampa IPTU (premium — dados exclusivos). Edge Function `seed-geosampa-iptu` implementa pipeline de ingestão: (a) Download CSV de IPTU do Portal de Dados Abertos SP (`dados.prefeitura.sp.gov.br`) — filtrado por distrito Moema, (b) Download shapefile de lotes fiscais do GeoSampa — filtrado por bounding box Moema, (c) JOIN por SQL (Setor-Quadra-Lote) para unir geometria + atributos, (d) Converte para GeoJSON, (e) Match com edifícios existentes via `ST_DWithin(30m)` ou JOIN por endereço normalizado. Campos enriquecidos: `edificios.total_units` (nº unidades autônomas), `edificios.area_construida` (área total m²), `edificios.ano_construcao` (ano), `edificios.padrao_iptu` (A/B/C/D/E → mapeado para alto/medio/popular), `edificios.tipo_uso_iptu` (residencial/comercial/misto), `edificios.num_pavimentos`, `edificios.sql_lote` (chave GeoSampa). Campo `edificios.seed_source_secondary = 'geosampa_iptu'`. Mapeamento padrão: A/B→alto, C→medio, D/E→popular.
- [ ] AC5: **Dados avançados NUNCA sobrescrevem input manual.** Regra de merge: se campo já preenchido manualmente pela consultora (`seed_source` != fonte automática ou campo editado manualmente), preservar valor manual. Lógica: `UPDATE edificios SET campo = COALESCE(campo, novo_valor)` — só preenche se NULL. Se consultora editou manualmente (field-level tracking via `edited_fields JSONB[]`), NUNCA sobrescreve. Princípio: dado humano > dado automático.
- [ ] AC6: Dashboard de status do pré-mapeamento. Na tela do epicentro/raio, seção "Dados Avançados" exibe: (a) progresso: "Edifícios enriquecidos: [N]/[total] ([%])", (b) breakdown por fonte: "Google Places: [N] | OSM: [N] | GeoSampa IPTU: [N] | Manual: [N]", (c) campos preenchidos: barra de progresso por campo (unidades: [%], padrão: [%], ano: [%]), (d) botão "Atualizar Dados" para forçar re-execução manual. Dados de `edificios` aggregate por `seed_source_secondary`.
- [ ] AC7: Resiliência e retry para fontes externas. Cada fonte executa independentemente — falha de uma não bloqueia as outras. Se Google Places API falha: retry 3x com backoff exponencial (1s, 5s, 15s). Se GeoSampa indisponível: skip, log warning, retry no próximo trigger. Se OSM Overpass timeout: reduzir raio de busca pela metade, retry. Todas as falhas logadas em `intelligence_feed` tipo `seed_completo` com metadata `{fonte, status: 'failed', erro}`. Status de cada fonte visível no dashboard (AC6).

## Technical Notes

- **Database:** UPDATE em `edificios` para enriquecimento (total_units, area_construida, ano_construcao, padrao_iptu, tipo_uso_iptu, num_pavimentos, sql_lote, seed_source_secondary). INSERT para novos edifícios descobertos. COALESCE para merge non-destructive. Campo `edited_fields JSONB` para tracking de edições manuais.
- **Google Places API:** `nearbysearch` com `type=apartment_building`. Rate limit: 1000 req/dia (free tier), 50 req/s. Key em Supabase secrets `GOOGLE_API_KEY`. Pagination: `next_page_token` para resultados > 20.
- **OSM Overpass:** `https://overpass-api.de/api/interpreter` com query Overpass QL. Sem autenticação. Rate limit informal (uso justo). Timeout: 180s.
- **GeoSampa/IPTU:** Pipeline ETL conforme `docs/research/geosampa-iptu-verification.md` seção 7. CSV download + shapefile + JOIN por SQL. Processamento via Edge Function ou script Node.js. Shapefile parsing via `shapefile` npm package ou `ogr2ogr` em container.
- **Triggers:** Database webhook on UPDATE `epicentros` WHERE `raio_ativo_m` changed — invoca Edge Function.
- **Components:** AdvancedSeedDashboard (status), SeedProgressBar (progresso por fonte/campo), SeedForceUpdate (botão atualizar).

## Notes

- **Princípio PV — Dados manuais > dados automáticos:** Input manual da consultora NUNCA é sobrescrito por dados automáticos. COALESCE preenche apenas campos NULL. Edições manuais são preservadas via field-level tracking.
- **Princípio PV — Graceful degradation:** Se todas as fontes falharem, edifícios continuam com dados base (OSM seed da Story 1.7). Pré-mapeamento avançado é enriquecimento, não dependência.
- **Research — OSM Overpass:** 16.595 buildings verificados para Moema (raio 2km). 7% com nome, 7% com endereço, 0% com nº unidades. Referência: `docs/research/google-places-verification.md`.
- **Research — Google Places:** Estimativa 100-400 edifícios residenciais em Moema. Usar `type=apartment_building` (NOT `premise` que retorna comércios misturados). Referência: `docs/research/google-places-verification.md`.
- **Research — GeoSampa IPTU:** Fonte premium com dados exclusivos — nº unidades, área construída, padrão construtivo (A-E), ano construção. Nenhuma outra fonte gratuita fornece estes dados. Pipeline ETL: CSV IPTU + shapefile lotes + JOIN por SQL (Setor-Quadra-Lote). Referência: `docs/research/geosampa-iptu-verification.md`.
- GeoSampa IPTU é Must para esta story — o salto qualitativo de 0% de dados de unidades (OSM) para ~100% (IPTU) é o principal valor desta story.
- Dados IPTU são estáticos (atualização anual) — pipeline de batch download + processamento offline é mais robusto que API em tempo real.
