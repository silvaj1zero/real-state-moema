# Story 1.6: Suporte Offline e Sincronização

**Epic:** 1 — Foundation, Mapa & Registro de Campo
**Status:** Ready
**Priority:** Must
**Estimated Complexity:** L

## User Story

Como consultora RE/MAX,
Quero cadastrar e qualificar edifícios mesmo sem internet,
Para que eu não perca dados em áreas de sinal fraco entre prédios altos de Moema.

## Acceptance Criteria

- [ ] AC1: Service Worker (Workbox 7.x) cacheia assets completos do App Shell via `precacheAndRoute`: HTML, JS, CSS, fontes, ícones (~2 MB). Estratégia `CacheFirst` para fontes (cache `fonts-cache`, TTL 1 ano, ~500 KB) e imagens estáticas (cache `images-cache`, TTL 30 dias, ~1 MB).
- [ ] AC2: Map tiles da região do epicentro pré-cacheados via Service Worker. Estratégia `CacheFirst` (cache `map-tiles-cache`, TTL 7 dias). Bounding box: epicentro + 2.5km em cada direção. Zoom levels 12-17 (~55 MB estimado). Pre-cache trigger: ao configurar epicentro ou sob demanda via botão "Baixar mapa offline" em Configurações.
- [ ] AC3: Cadastro de edifício offline salvo em IndexedDB (store `mutation-queue`). Mutation com: id (UUID gerado no client via `crypto.randomUUID()`), table ('edificios' + 'edificios_qualificacoes'), operation ('INSERT'), payload (todos os campos), created_at (ISO timestamp), retry_count (0), status ('pending'). Edifício também salvo no cache local (store `edificios-cache`) para exibição imediata.
- [ ] AC4: Indicador visual "Offline" no header do mapa conforme frontend-spec: badge laranja à esquerda quando `navigator.onLine === false`. Transição animada ao mudar de online para offline e vice-versa.
- [ ] AC5: Sincronização automática ao reconectar. Background Sync API (`registration.sync.register('sync-mutations')`) replay mutations na ordem FIFO. Fallback: polling a cada 30s se Background Sync API indisponível (`!('SyncManager' in window)`). Durante sync: badge amarelo "Sincronizando..." com contagem de pendentes.
- [ ] AC6: Edifícios cadastrados offline exibem badge visual "pendente sync" (ícone nuvem com seta) no pin do mapa e no card. Após sync bem-sucedido: badge removido, toast "N edifícios sincronizados". Se sync falhar após 5 retries: badge erro (ícone vermelho), notificação ao usuário.
- [ ] AC7: Conflito de dados resolvido por Last-Write-Wins (LWW) baseado em `updated_at`. Server version prevalece em conflito. Dado local atualizado com versão do servidor após sync. Adequado para MVP single-user. Log de conflitos no console para debugging.
- [ ] AC8: Mapa navegável offline na área pré-cacheada. Pins de edifícios exibidos a partir do cache IndexedDB (store `edificios-cache`). Interações disponíveis offline: ver mapa, ver pins, abrir card (dados em cache), cadastrar edifício, qualificar edifício, alterar status.
- [ ] AC9: API responses cacheadas via `StaleWhileRevalidate` (cache `api-cache`). Dados de edifícios e qualificações do Supabase salvos em IndexedDB stores (`edificios-cache`, `qualificacoes-cache`) com indexes para queries locais. Atualização do cache em background quando online.

## Technical Notes

- **Database:** Mutation queue no IndexedDB (store `mutation-queue`) com schema: `{ id: string, table: string, operation: 'INSERT'|'UPDATE'|'DELETE', payload: Record<string, unknown>, created_at: string, retry_count: number, status: 'pending'|'syncing'|'failed' }`. Cache stores: `edificios-cache` (key: edificio.id, value: Edificio, indexes: coordinates), `qualificacoes-cache` (key: id, value: EdificioQualificacao, indexes: edificio_id), `leads-cache` (key: id, value: Lead, indexes: edificio_id, etapa_funil), `settings` (key-value).
- **API:** Sync function replays mutations via Supabase client SDK na ordem FIFO. Cada mutation é uma operação Supabase (insert/update/delete). Background Sync Event: `sync-mutations`.
- **UI Components:** OfflineBadge (header, laranja), SyncStatusBadge (header, amarelo/verde/vermelho), PendingSyncBadge (pin badge, nuvem com seta), SyncErrorBadge (pin badge, vermelho), OfflineMapSettings (botão "Baixar mapa offline"), Toast de sync.
- **PostGIS:** Queries offline não usam PostGIS (server-side). Filtragem local de edifícios por coordenadas via Turf.js `booleanPointInPolygon` ou cálculo de distância simples no client.
- **Offline:** Esta é a story principal de offline. Usa Workbox para Service Worker, `idb-keyval` para operações simples, `idb` wrapper para stores estruturadas. Workbox Background Sync Plugin para retry automático.

## Notes

- **Princípio PV — Funciona sozinho antes de depender de externo:** O app deve ser plenamente funcional offline para operações de campo. Conexão é necessária apenas para sync e funcionalidades que dependem de APIs externas (geocoding, seed data).
- **Estimativa de storage:** ~55 MB para tiles + ~2 MB para App Shell + ~500 KB para dados em IndexedDB = ~58 MB total. Browsers modernos permitem até 50% do disco disponível. Alertar se storage quota estiver baixo.
- O pre-cache de tiles pode demorar em conexões lentas. Implementar progress bar e permitir cancelamento. Sugerir fazer download em WiFi.
- Considerar usar `navigator.storage.persist()` para evitar que o browser limpe o cache automaticamente.
- Para operações de qualificação offline (alterar status, notas, abertura), o mesmo padrão de mutation queue se aplica. Toda write operation passa pela queue quando offline.
- **Testing:** Testar com DevTools Network offline, simular reconexão, verificar order de mutations, testar conflitos com dados alterados no servidor durante offline.
