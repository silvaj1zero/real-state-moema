## User Story

Como sistema de prospeccao,
Quero ter schemas Zod unificados (Advertiser, Property, HomeFlags) compartilhados entre crawlers e API,
Para que classificacao, persistencia e validacao tenham fonte unica de verdade e tipagem ponta-a-ponta.

## Dev Agent Record

**Mode:** Interactive

**Start/End:** 2026-05-18 (sessao unica ~2h, schemas + testes + migration + barrel)

**Decisions:**
- Mantida superficie publica EXCLUSIVAMENTE via barrel `index.ts`. Stories 7.2-7.9 importam de `app/src/lib/schemas/epic7` (sem `/agent`, `/broker` etc.) para preservar refactors internos sem quebrar consumers. Documentado no JSDoc do barrel.
- Pureza TS forcada: nenhum arquivo do pacote `epic7/` importa `next/*`, `@/lib/supabase/*` ou Node built-ins. Decisao garante copy-on-build em Apify Actors (Story 7.4) e Supabase Edge Functions (Story 7.7) sem rewrite.
- Slots Agent/Broker/Builder/Office sao todos `.nullable()` (nao `.optional()`). Motivo: HomeHarvest upstream emite `null` explicito quando ausente, e Zod `.nullable()` aceita `null` (diferente de `.optional()` que aceita `undefined`). Match exato com payload do crawler.
- `ClassificationSignalSchema` modelado como `z.enum([...])` (vocabulario fechado) em vez de `z.string()` livre. Forca Story 7.3 a usar sinais pre-aprovados; novos sinais entram via edicao explicita do schema.
- CRECI default `false` no `creci_validated` ao inves de `null` — semantica conservadora (so `true` apos buscacreci confirmar). Alinhado com pattern HomeFlags.
- Migration 008 usa `DEFAULT 'unknown'` na ALTER scraped_listings.classification: dados Epic 6 ja existentes nao violam CHECK no momento do apply (mitiga risco listado).
- Index parcial em `leads.lead_type` (`WHERE lead_type IS NOT NULL AND lead_type <> 'UNKNOWN'`): evita inchaco do indice com a massa de leads ainda nao classificados (90%+ ate Story 7.3 rodar).

**Surprises/Gotchas:**
