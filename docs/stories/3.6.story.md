# Story 3.6: Cross-Referencing entre Portais

**Epic:** 3 — Inteligência, ACM & Agentes de Automação
**Status:** Ready
**Priority:** Should
**Estimated Complexity:** L

## User Story

Como consultora RE/MAX,
Quero que o sistema cruze anúncios entre portais para consolidar duplicatas, detectar transições "ex-imobiliária para FISBO" e calcular tempo de mercado,
Para que eu tenha inteligência acionável sobre proprietários motivados e tendências de preço na minha micro-região.

## Acceptance Criteria

- [ ] AC1: Consolidação de duplicatas entre portais. Algoritmo de deduplicação: para cada `scraped_listing` ativo, busca candidatos de OUTROS portais com `ST_DWithin(coordinates, 30m)` AND `ABS(area_m2 - area_m2) < 5` AND `ABS(preco - preco) / preco < 0.05` (mesma localização, área similar +-5m², preço similar +-5%). Match score: 0-100 baseado em proximidade + similaridade de atributos. Score >= 80: auto-merge. Score 60-79: sugestão para revisão manual. Score < 60: não merge. Resultado: campo `scraped_listings.merged_group_id UUID` agrupa anúncios do mesmo imóvel. Tabela auxiliar `listing_cross_refs` (id, listing_a_id, listing_b_id, match_score, merged_at, reviewed_by) para tracking.
- [ ] AC2: Detecção de transição "ex-imobiliária para FISBO". Quando anúncio com `tipo_anunciante = 'imobiliaria'` desaparece (`is_active = false`) e novo anúncio com `tipo_anunciante = 'proprietario'` aparece no mesmo endereço (`ST_DWithin 50m`) dentro de 30 dias: evento `intelligence_feed` tipo `ex_imobiliaria_fisbo` com prioridade `alta`. Título: "Possível ex-imobiliária → FISBO: [endereço]". Descrição: "Anúncio de imobiliária removido em [data]. Novo anúncio de proprietário em [data]. Oportunidade: proprietário frustrado com imobiliária anterior." Metadata JSONB: `{listing_antigo_id, listing_novo_id, imobiliaria_anterior, dias_entre}`. Este é um dos sinais mais valiosos — proprietário insatisfeito com imobiliária está aberto a nova abordagem.
- [ ] AC3: Cálculo de tempo de mercado. Para cada `scraped_listing` ativo: `tempo_mercado_dias = CURRENT_DATE - first_seen_at::date`. Para listings com `merged_group_id`: usar `MIN(first_seen_at)` do grupo (primeiro anúncio). Exibido no card do listing: "No mercado há [N] dias" com cores: verde <= 30 dias, amarelo 31-90 dias, vermelho > 90 dias. Badge no mapa para listings com > 90 dias: ícone relógio vermelho. Tempo de mercado longo = proprietário cada vez mais motivado = oportunidade.
- [ ] AC4: Tracking de mudança de preço consolidada. Para listings no mesmo `merged_group_id`: construir histórico de preço unificado. Timeline visual: "[data] R$ [preco] (ZAP) → [data] R$ [preco] (OLX) → [data] R$ [preco] (ZAP — redução 10%)". Exibido no detalhe do listing como gráfico sparkline de preço ao longo do tempo. Reduções de preço destacadas com seta vermelha e %. Dados alimentam ACM (Story 3.1): histórico de preço = tendência de mercado.
- [ ] AC5: Alimentação da ACM com dados cross-referenciados. Quando cross-referencing consolida dados de múltiplos portais: (a) preço mais recente é usado como referência, (b) tempo de mercado longo + reduções = indicador de preço acima do mercado, (c) dados consolidados disponíveis para importação na ACM (Story 3.1, AC6) com fonte "Scraping (consolidado)" e metadata com histórico. INSERT/UPDATE em `acm_comparaveis` com `scraped_listing_id` apontando para listing principal do merged group.
- [ ] AC6: Execução do cross-referencing como job periódico. Supabase Edge Function `cross-reference-listings` executada: (a) após cada varredura de portal (Story 3.4, AC5 — chain), (b) via cron semanal (domingo 03:00 BRT) para reconciliação completa. Processo: buscar todos listings ativos, executar deduplicação (AC1), detectar transições (AC2), calcular tempos (AC3), consolidar preços (AC4). Logging em `intelligence_feed` tipo `sync_completo` com metadata cross-ref stats.
- [ ] AC7: Tela de revisão de cross-references. Na tela "Agentes" (Story 3.4, AC9), aba "Cross-Refs" exibe: (a) grupos consolidados com score, (b) sugestões pendentes de revisão (score 60-79) com botões "Confirmar merge" / "Rejeitar", (c) transições ex-imobiliária→FISBO detectadas com link para lead/edifício, (d) métricas: total grupos, total transições detectadas, média tempo de mercado. Filtros: por portal, por score, por tipo de evento.

## Technical Notes

- **Database:** Tabela auxiliar `listing_cross_refs` (id UUID PK, listing_a_id UUID FK, listing_b_id UUID FK, match_score INTEGER, is_confirmed BOOLEAN, merged_at TIMESTAMPTZ, reviewed_by UUID). Campo `scraped_listings.merged_group_id UUID` para agrupamento. Queries PostGIS para `ST_DWithin(30m)` cross-portal. Intelligence feed events para transições detectadas.
- **Deduplicação:** Algoritmo em Edge Function: (1) buscar listings ativos agrupados por proximidade geográfica (ST_ClusterDBSCAN com eps=30m), (2) dentro de cada cluster, comparar atributos (área, preço), (3) calcular score, (4) merge automático ou sugestão. Complexidade: O(n log n) com spatial index.
- **API:** Supabase client SDK. TanStack Query key: `['cross-refs']`, `['cross-refs', 'pending']`. Mutation para confirmar/rejeitar merge.
- **Components:** CrossRefReviewPanel (tela revisão), CrossRefGroup (grupo consolidado), CrossRefSuggestion (sugestão pendente), ListingTimelineChart (sparkline preço), TransitionAlert (alerta ex-imob→FISBO).

## Notes

- **Princípio PV — Scraping é acelerador, não dependência:** Cross-referencing só processa dados que existem. Se poucos portais varridos, algoritmo funciona com o que há. Se nenhum dado, seção não aparece.
- **Princípio PV — Graceful degradation:** Se cross-referencing falha, listings individuais continuam disponíveis sem consolidação. Nenhuma funcionalidade depende de cross-refs para operar.
- A detecção "ex-imobiliária→FISBO" é um dos sinais mais valiosos do sistema — proprietário insatisfeito com imobiliária anterior é altamente receptivo a abordagem profissional da Luciana.
- Tempo de mercado > 90 dias é forte indicador de proprietário motivado — preço provavelmente acima do mercado, abertura para negociação.
- Match score conservador (>=80 para auto-merge) minimiza falsos positivos. Sugestões (60-79) requerem validação humana.
- Histórico de preço consolidado alimenta tendência na ACM (Story 3.1, AC3) — dado valioso para precificação.
