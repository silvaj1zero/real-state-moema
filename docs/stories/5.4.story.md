# Story 5.4: Error Propagation — Eliminar Falhas Silenciosas

**Epic:** 5 — Correção de Divida Tecnica (Audit PV)
**Status:** Done
**Priority:** Should
**Estimated Complexity:** M
**Audit Finding:** F6 (MEDIUM)

## User Story

Como consultora RE/MAX usando o app em campo,
Quero ver uma mensagem clara de erro quando o sistema falha ao carregar dados,
Para que eu saiba que ha um problema de conexao e nao ache que "nao tenho leads" quando na verdade o servidor esta fora.

## Acceptance Criteria

- [x] AC1: Auditar TODOS os hooks que fazem queries Supabase. Identificar pattern de error swallowing: `if (error) { console.error(...); return [] }`. Listar ocorrencias. Hooks conhecidos: `useLeadsByEdificio`, `useLeadsByFunnel` (useLeads.ts:38-40, 76-80).
- [x] AC2: Para cada hook identificado no AC1: substituir `return []` por `throw new Error(...)` dentro do `queryFn`. TanStack Query captura o erro e expoe via `query.error`. Componentes que consomem o hook ja recebem `error` no retorno — garantir que esta sendo propagado.
- [x] AC3: Criar componente `ErrorBanner` reutilizavel: banner vermelho no topo da lista/tela com mensagem "Erro ao carregar dados. Verifique sua conexao." + botao "Tentar novamente" que chama `refetch()`. Componente recebe `error` e `onRetry` como props.
- [x] AC4: Implementar `ErrorBanner` nas telas criticas: Dashboard, Safari (mapa), Funil, Feed, ACM. Se `query.error` existe, exibir banner em vez de lista vazia.
- [x] AC5: Manter retorno de `[]` como fallback APENAS para `query.data` (quando loading ou sem dados reais), NAO para erro. Diferenca: `data ?? []` (ok) vs `if (error) return []` (proibido).
- [x] AC6: Testes para cada hook corrigido: simular erro de Supabase (mock reject), verificar que `error` e propagado e `data` e undefined (nao `[]`).

## Technical Notes

- **Pattern atual (ERRADO):**
  ```typescript
  if (error) { console.error('...', error); return [] }
  ```
- **Pattern correto:**
  ```typescript
  if (error) throw new Error(`Failed to fetch: ${error.message}`)
  ```
- **TanStack Query:** Quando `queryFn` throws, o query entra em estado `error` e `query.error` contem o erro. Retry automatico (3x por default) antes de mostrar erro final.
- **ErrorBanner:** Componente simples, sem logica complexa. Nao precisa de store.
