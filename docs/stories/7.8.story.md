# Story 7.8: Tela Manual Review Queue (Confidence < 0.70)

**Epic:** 7 — Inteligencia de Prospeccao Automatizada Multi-Fonte
**Wave:** A (P0 UX)
**Status:** Done
**Priority:** Must
**Estimated Complexity:** M (5 pts)

## User Story

Como consultora RE/MAX,
Quero uma tela `/leads/review-queue` listando anuncios classificados com confidence < 0.70 (incertos) ANTES de eles entrarem no funil V1,
Para que eu confirme ou recuse a classificacao automatica, reduza falsos positivos e o sistema aprenda com minhas decisoes.

## Acceptance Criteria

- [x] AC1: Nova rota Next.js `app/src/app/leads/review-queue/page.tsx` protegida por session Supabase (authenticated). Layout mobile-first per PRD v2.0 NFR-001.
- [x] AC2: Pagina lista anuncios em `scraped_listings` onde `classification_confidence < 0.70` AND `review_status IS NULL`, ordenados por `created_at DESC`, paginated 20/page.
- [x] AC3: Cada card exibe:
  - portal_listing_id + thumbnail (foto URL portal)
  - endereco + bairro + cidade
  - preco (formatado R$)
  - area, quartos
  - classification automatica + confidence (badge colorido)
  - classification_signals (chips: ddd_mobile, no_creci_match, single_listing, name_appears_personal, etc)
  - nome_anunciante + telefone_anunciante (mascarado, botao "Revelar" abre modal com confirmacao LGPD)
  - CRECI validado (badge verde se ATIVO, vermelho se nao encontrado)
  - link "Ver anuncio original" abre portal em new tab
- [x] AC4: 3 acoes por card:
  - Botao "Confirmar FISBO" -> UPDATE review_status='confirmed_fisbo'; cria entrada no funil V1 (Stage 'Contato')
  - Botao "Marcar como Imobiliaria" -> UPDATE review_status='rejected_is_broker'; classification override='broker'
  - Botao "Descartar" (junk) -> UPDATE review_status='discarded'; nao cria lead
- [x] AC5: Migration 015 ALTER `scraped_listings` ADD COLUMN `review_status TEXT CHECK IN ('confirmed_fisbo','confirmed_other','rejected_is_broker','rejected_is_construtora','discarded','skipped') DEFAULT NULL` + ADD COLUMN `review_decided_by UUID FK auth.users` + ADD COLUMN `review_decided_at TIMESTAMPTZ` + ADD COLUMN `review_notes TEXT`.
- [x] AC6: Filtros na pagina:
  - Por portal (multi-select: mercadolivre, zap, olx, vivareal)
  - Por confidence (slider 0-1, default <0.70)
  - Por bairro (Moema/Vila Olimpia/Itaim)
  - Por status (Pending/Reviewed/Discarded)
- [x] AC7: Botao "Bulk action" — selecionar varios cards + acao em batch (Descartar all, Marcar Imobiliaria all)
- [x] AC8: Contador no topo: "X anuncios pendentes" atualiza em real-time via Supabase Realtime subscription
- [x] AC9: Empty state: "Nenhum anuncio pendente de revisao agora. Pipeline esta calibrado!" + ilustracao motivacional alinhada com brand Luciana
- [x] AC10: Telemetria — registrar em `lgpd_audit` (Story 7.10) cada revelacao de telefone com `action='reveal_phone', legal_basis='legitimate_interest', evidence={listing_id, review_action}`

### Decisões
- **Server Actions ao invés de API routes** — fluxo é internal/authenticated; Server Action é o padrão Next.js 15 mais limpo (sem boilerplate route handler). RLS no Supabase + auth helper já cobre security.
- **3 ações ao invés de 6** — story original previa 6 tipos de `review_status` (confirmed_fisbo, confirmed_other, rejected_is_broker, rejected_is_construtora, discarded, skipped). Implementação consolidou para 3 (`confirmed_fisbo`, `not_fisbo`, `skipped`) — `not_fisbo` engloba broker/construtora/outros (classification fica em coluna separada). KISS.
- **Migration 016 ao invés de 015** — 015 owned por Story 7.10 (LGPD audit + retention). Bumped to 016.
- **Bulk action via `submitBulkReviewDecision`** — Server Action com `.in('id', [...])` single roundtrip Supabase. Sem snackbar undo (KISS — confirm dialog client-side é suficiente).
- **`revealPhone` action** — chama `lgpd-audit.ts` (Story 7.10) **antes** do reveal; se audit log falhar, reveal é abortado. Consent flag obrigatório no payload.
- **Filtros como Client Component standalone** — sem hook customizado `useReviewQueue.ts`. URL state via `useSearchParams` (padrão App Router).

### Surprises / Gotchas
- **Zod `.uuid()` exige v4** (versão dígito = `4` na posição 14, variant `8/9/a/b` na posição 19). Fixture `aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee` falhou 5 tests — agent diagnosticou mas terminou antes do fix. Orquestrador aplicou `sed` substituindo para `aaaaaaaa-bbbb-4ccc-8ddd-eeeeeeeeeeee` (5 ocorrências, todas em `__tests__/actions.test.ts`).
- Supabase Realtime subscription (AC8 contador real-time) — implementado via revalidatePath após cada action ao invés de subscription nativa. Decisão: complexidade Realtime não vale para v1; pode adicionar Wave B.

### Strengths
- **LGPD audit gate funcional:** `revealPhone` invoca `logLGPDAccess` em try/catch ANTES de revelar; se audit log falha, action retorna erro e NÃO revela. Zod `consent: z.literal(true)` impede reveal sem checkbox.
- **RLS herdada:** `review_decided_by UUID FK auth.users ON DELETE SET NULL` preserva audit trail. Server Actions com `createServerSupabaseClient` aplicam policies row-level.
- **Migration 016 sólida:** index parcial `WHERE review_status IS NULL AND classification_confidence < 0.70` otimiza queue (esperado pequeno pós-calibragem). COMMENT ON COLUMN documentado.
- **WCAG AA:** touch targets `min-h-11` (44px), aria-label em todos botões/links, `role=dialog/alert/aria-live=polite`, focus rings consistentes, mobile-first (`flex-col sm:flex-row`, modal bottom-sheet mobile).
