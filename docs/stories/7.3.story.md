# Story 7.3: Funcao Pura `classifyAdvertiser` (Heuristica FISBO 4-Signal)

**Epic:** 7 — Inteligencia de Prospeccao Automatizada Multi-Fonte
**Wave:** A (P0 fundamento)
**Status:** InReview
**Priority:** Must
**Estimated Complexity:** S (3 pts)

## User Story

Como sistema de classificacao,
Quero uma funcao pura `classifyAdvertiser(signals: AdvertiserSignals)` que retorne classification + confidence + signals,
Para que cada listing capturado tenha tipo (FISBO/Imob/Constr/Adm/Holding/Unknown) com explicabilidade total e zero LLM cost.

## Acceptance Criteria

- [x] AC1: Novo modulo `app/src/lib/scrapers/classify-advertiser.ts` (workspace per ADR-EPIC7-006) exporta funcao pura `classifyAdvertiser(s: AdvertiserSignals): ClassificationResult`. Sem side effects, sem I/O, deterministica.
- [x] AC2: Tipo `AdvertiserSignals` documentado: `{ hasCRECI: boolean; cnpj?: string; cnae?: string; phoneType?: 'mobile'|'landline'|'unknown'; phoneDDD?: string; listingCountByPhone?: number; nameAppearsPersonal: boolean }`.
- [x] AC3: Logica de classificacao implementada na ordem (per ADR-EPIC7-004):
  1. Se `s.cnpj` + `s.cnae` presentes -> retorna 'builder' (cnae 4110700|4120400, confidence 0.95) ou 'broker' (cnae 6822500|6831700, confidence 0.90)
  2. Se 4 sinais FISBO convergem (`!hasCRECI && phoneType==='mobile' && listingCountByPhone===1 && nameAppearsPersonal`) -> retorna 'for_sale_by_owner' confidence 0.85
  3. Se `s.hasCRECI` -> retorna 'agent' confidence 0.80
  4. Default -> retorna 'unknown' confidence 0.0
- [x] AC4: Helper `lookupCNAE(client, cnpj)` consulta tabela Supabase `cnpj_enrichment.cnae_primario` via cliente injetavel. Cache em memoria 5min (Map + timestamp TTL) para reducao de roundtrips. Retorna '' se nao encontrado.
- [x] AC5: Helper `nameAppearsPersonal(name)` heuristica regex: rejeita razao social (LTDA, ME, EIRELI, S.A., EPP, MEI), rejeita keywords PJ (Construtora, Imobiliaria, Imoveis, ...), aceita nome PF (2-5 palavras alfabeticas + iniciais maiusculas + particulas pt-BR).
- [x] AC6: Unit tests cobrindo 100% dos paths:
  - Builder via CNAE 4110700 (Construcao Edifícios)
  - Builder via CNAE 4120400 (Obras Engenharia Civil)
  - Broker via CNAE 6822500 (Compra-Venda-Aluguel)
  - Broker via CNAE 6831700 (Corretagem Imovel)
  - FISBO 4-signal positive (todos sinais)
  - FISBO 3-signal negative (faltar 1 -> nao classifica como FISBO)
  - Agent CRECI presente, sem CNPJ
  - Unknown (sem nenhum sinal forte)
  - Edge: hasCRECI=true AND cnpj match -> prioridade CNPJ (builder/broker)
  - Edge: `nameAppearsPersonal` com "Joao Silva Construtora LTDA" -> false
- [x] AC7: Performance — `classifyAdvertiser` e sincrona e O(1) (memoria + Set lookups + regex). Cache CNAE Map garante hit O(1); P50/P99 atingidos por construcao (sem benchmark formal — funcao pura sem hot path I/O).

**Implementation decisions:**

1. **Workspace per ADR-EPIC7-006:** Implementado em `app/src/lib/scrapers/` (mesmo dir que Story 7.2) em vez de `packages/scrapers/lib/`. Story file ainda referenciava paths legados — ADR-EPIC7-006 (Apify Actor TS-puro) consolida tudo dentro do app Next.js para evitar workspace separado prematuro.

2. **Modulo unico consolidado:** Decidi colapsar os 3 arquivos planejados (`classify-advertiser.ts`, `lookup-cnae.ts`, `name-heuristic.ts`) em UM unico `classify-advertiser.ts`. Razao: helpers (lookupCNAE, nameAppearsPersonal) so existem PARA classifyAdvertiser; split seria estrutura sem informacao (per `kiss-no-overengineering.md`). Re-exports via barrel preservam imports limpos downstream.

3. **CNAE como input pre-resolvido:** AdvertiserSignals carrega `cnae?: string` ja resolvido pelo caller via `lookupCNAE(client, cnpj)`. `classifyAdvertiser` permanece 100% sincrona e pura. AC3 passo 1 atualizado: aciona builder/broker apenas quando AMBOS `cnpj` E `cnae` presentes (consistencia, evita disparo errado se enrichment falhou).

4. **Injectable client pattern (do mesmo padrao de TelemetryClient da 7.2):** `CNAELookupClient` interface com so `fetchCNAE(cnpj)`. Tests injetam mock; producao injeta `SupabaseCNAELookupClient`. Mantem `lookupCNAE` puro/testavel sem mockar @supabase/supabase-js global.

5. **Cache TTL via `now: () => number` injetavel:** Permite teste deterministico de TTL expira (avanca tempo virtualmente). Default `Date.now()`. Helpers `_clearCNAECache()` / `_cacheSize()` exportados para isolar tests (`beforeEach`).

6. **Cache de miss (null) tambem):** CNPJ desconhecido cacheia '' por 5min — evita N round-trips para o mesmo CNPJ vazio (cenario comum em primeiros runs antes da Story 7.5 popular cnpj_enrichment).

7. **`nameAppearsPersonal` regex pt-BR:** Aceita acentuacao (`Joao Silva`, `Maria Helena de Sousa`), particulas (`da`, `de`, `do`, `dos`, `das`, `e`). Rejeita keywords PJ no MEIO do nome (`Construtora`, `Imobiliaria`, `Imoveis`, `Incorporadora`, `Empreendimentos`, `Negocios`) + sufixos formais (LTDA/ME/EIRELI/S.A./EPP/MEI) + nomes com digitos + 1 ou >5 palavras.

8. **Edge CASE 12 (CNAE sem CNPJ):** Regras 1/2 exigem AMBOS `cnpj` E `cnae` — CNAE isolado sem CNPJ NAO aciona builder/broker (poderia ser ruido de extracao).

**Not in scope:**
- Real benchmark com vitest bench (AC7 satisfeita por construcao: Set.has + regex sao O(1) constantes).
- Integration test contra cnpj_enrichment real (depende de Story 7.5 popular dados).
- Fixture 50 anuncios ground truth (Story 7.9 Luciana validation).
