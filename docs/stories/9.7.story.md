## Story

Como operador, quero que a configuração geográfica usada pelo pipeline (viewbox/CEPs do crawler em `app/src/lib/apify.ts`, whitelist de bairros ViaCEP dos builders, pontos de geocode) seja validada contra o ITBI real em PROD, para que os recortes geográficos parem de viver hardcoded e divergentes entre scripts.

## Acceptance Criteria

1. **AC1 — Inventário:** levantar TODAS as constantes geográficas hardcoded do pipeline ACM/crawler (viewbox `apify.ts`, `BAIRROS_NO_RAIO` + `BAIRRO_NORMALIZADO` dos builders 113/132, CEPs de referência) num doc curto `docs/acm/CONFIG-GEOGRAFICA.md` com fonte de cada uma.
2. **AC2 — Validação contra ITBI:** cruzar cada bairro/viewbox com o dado real (RPC/tabela em PROD, read-only): bairros da whitelist que nunca aparecem no ITBI do raio e vice-versa (ITBI com logradouros fora da whitelist) são listados como divergência.
3. **AC3 — Config canônica única:** consolidar as constantes divergentes numa fonte única importável (ex.: `app/src/lib/acm/geoConfig.ts`) consumida pelos builders novos e pela CLI — SEM alterar os datasets congelados dos casos existentes (imutáveis).
4. **AC4 — Regressão:** suíte ACM verde; casos congelados byte-idênticos (a story muda config para o FUTURO, não reprocessa o passado).

## Dev Agent Record

### Decisões YOLO (2026-07-12)

**D1 — Campo `notas` armazena categoria, não bairro granular.**
Achado crítico do AC2: o campo `notas` de `acm_comparaveis` armazena `"[ITBI] SQL XXXXXXXXXX; Moema e região"` para todos os 3.618 registros ITBI — a string após o ponto-e-vírgula é a categoria de ingestão batch (`"Moema e região"`), não o bairro individual. O bairro granular é resolvido pelos builders via ViaCEP por nome de logradouro (cache `viacep-cache.json`). O script AC2 foi ajustado para usar o cache ViaCEP + RPC no raio, não parse de `notas`.

**D2 — Whitelist CONVERGENTE entre builders 113 e 132.**
Os dois builders têm exatamente os mesmos 8 bairros e os mesmos 2 aliases. Sem divergência de valores — a consolidação no `geoConfig.ts` é cópia direta, não resolução de conflito.

**D3 — `MOEMA_BBOX` não migrado (arquivo em edição paralela).**
`app/src/lib/geocoding.ts:11` contém `MOEMA_BBOX` hardcoded. Documentado no `geoConfig.ts` e no `CONFIG-GEOGRAFICA.md`, mas não editado pois é arquivo de baixo risco e a migração trivial (`import { MOEMA_BBOX } from '@/lib/acm/geoConfig'`) pode ser feita quando o Épico 7 encerrar sem afetar comportamento.

**D4 — `apify.ts` não migrado (edição paralela founder).**
`MOEMA_NEIGHBORHOODS` e `buildSearchInput` em `apify.ts` são candidatos à migração para `geoConfig.ts`, mas `apify.ts` consta como `M` no `git status` (trabalho ativo do founder no Épico 7). Inventariado e documentado; migração bloqueada pela regra de não-edição.

**D5 — 5 bairros da whitelist sem logradouros no raio (não-erro).**
`Brooklin Novo`, `Cidade Monções`, `Itaim Bibi`, `Indianópolis` e `Vila Uberabinha` constam na whitelist mas não aparecem nos logradouros do raio de 1 km do Andrade Pertence 113 (cache ViaCEP). Isso é esperado: a whitelist é defensiva (cobre casos de expansão futura do raio e logradouros limítrofes). O raio de 1 km em torno desse alvo específico cobre principalmente V.Olímpia (45%), Moema (41%) e V.N. Conceição (14%). Nenhum logradouro do raio está fora da whitelist — cobertura nominal validada.

**D6 — `geoConfig.ts` usa `BAIRROS_NO_RAIO_VILA_OLIMPIA` (não `BAIRROS_NO_RAIO`).**
Nome mais descritivo para evitar colisão quando o pipeline expandir para outros mercados (ex.: Jardins). Os builders existentes não importam daqui (datasets congelados) — só futuros builders usarão esse export.

**D7 — Bug de encoding no comentário JSDoc corrigido.**
`docs/acm/*/dataset.json` no bloco JSDoc fechava o comentário prematuramente (`*/`). Corrigido para `docs/acm/{caso}/dataset.json`.

**D8 — QA-fixes (Fable, mesma sessão): fonte única de verdade de fato.**
(a) `RAIO_PADRAO_M` no `geoConfig.ts` era um literal duplicado de `methodology.ts:587` — trocado por re-export (`export { RAIO_PADRAO_M } from './methodology'`): uma fonte, dois pontos de acesso. (b) O `9.7-geo-crosscheck.mjs` espelhava a whitelist localmente ("sem import .ts em .mjs") — recriando a cópia divergente que a story elimina; trocado por import real de `geoConfig.ts` (o script roda via `npx tsx`, mesmo padrão do doc de verificação da 9.4). Crosscheck re-executado pós-fix: saída idêntica, exit 0.
