# Story 7.13: Wiring ao vivo — publisher_type + proxy tiering no caminho ZAP/VivaReal

**Epic:** 7 — Inteligencia de Prospeccao Automatizada Multi-Fonte
**Wave:** A (integração)
**Status:** Done
**Priority:** Should (ativa o ganho real de 7.11 + 7.12 no fluxo de produção)
**Estimated Complexity:** S (3-5 pts)

## User Story

Como sistema de prospeccao,
Quero ligar as capacidades determinísticas de 7.11 (`publisher_type`) e 7.12 (proxy residencial por alvo) ao caminho de ingestão ao vivo de ZAP/VivaReal (busca paramétrica do Epic 6),
Para que o sinal nativo de anunciante seja persistido e os alvos Cloudflare usem proxy residencial — convertendo as libs já testadas em ganho real de produção.

## Contexto

As Stories 7.11 e 7.12 entregaram libs puras e testadas (`normalizePublisherType`, `resolveProxySpec`/`buildProxyConfiguration`, block-rate na telemetria), mas o **wiring ao vivo** ficou pendente (obs. low dos QA gates de ambas). O crawler MercadoLivre (Story 7.4) não se beneficia: ML não expõe `publisherType` nem é Cloudflare. O caminho que se beneficia é **ZAP/VivaReal**, ingerido via `apify-parametric.ts` (Epic 6) usando o actor `viralanalyzer/brazil-real-estate-scraper`.

- `normalizeListing` (apify.ts) já mapeia `raw.advertiserType → publisher_type` (7.11), mas `upsertListing` não persiste a coluna (que não existe).
- `buildParametricSearchInput` monta o INPUT do actor mas não envia `proxyConfiguration`.

## Acceptance Criteria

- [x] AC1: Migration adiciona coluna `scraped_listings.publisher_type` (text nullable, CHECK ∈ {owner,agency,developer} ou NULL). Aplicável via SQL Editor (scaffold Supabase dessincronizado — ver runbook).
- [x] AC2: `upsertListing` (apify-parametric.ts) persiste `publisher_type` no INSERT e atualiza no UPDATE (campo nativo determinístico de ZAP/VivaReal; NULL para fontes sem o campo).
- [x] AC3: `buildParametricSearchInput` injeta `proxyConfiguration` no INPUT do actor por alvo, via `resolveApifyInputProxy(portal)` — residencial BR para zap/vivareal, datacenter para olx (tiering da 7.12). Convenção Apify: `{ useApifyProxy, apifyProxyGroups, apifyProxyCountryCode }`.
- [x] AC4: `ScrapedListing` (types.ts) ganha `publisher_type: PublisherType | null`.
- [x] AC5: Testes: `resolveApifyInputProxy` (grupo/país por alvo + fallback); `buildParametricSearchInput` inclui proxyConfiguration correto por portal. Sem regressão no escopo scrapers/apify.
- [x] AC6: Fallback gracioso preservado (7.12): se residencial indisponível, o input cai para datacenter (mesma lógica de `resolveProxySpec`).
