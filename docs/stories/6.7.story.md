# Story 6.7: UI "Quem e o dono?" (Dossie de Proprietario)

**Epic:** 6 — Busca Parametrica On-Demand com Enriquecimento de Contatos
**Status:** Done
**Priority:** Should
**Estimated Complexity:** S

## User Story

Como consultora RE/MAX,
Quero clicar num botao "Quem e o dono?" dentro do card do edificio,
Para receber na mesma tela um dossie com o nome do proprietario, matricula e contatos ja enriquecidos.

## Acceptance Criteria

- [x] AC1: Botao "Quem e o dono?" no componente `BuildingCard` visivel apenas se `edificio.tipo_residencial IS TRUE`. Icone de pessoa + tooltip "Consulta cartorio (R$0,28)". Estados: `idle`, `loading`, `success`, `error`, `forbidden` (rate limit) e `budget_exceeded`.
- [x] AC2: Ao clicar, abre `OwnerLookupModal`: estado inicial mostra spinner + "Consultando cartorio..." e dispara `useOwnerLookup.mutate({ edificio_id })`. Se ja houver cache (`cache_hit=true`), modal nao chama API e mostra payload existente imediatamente com badge "Dados de cache (X dias)".
- [x] AC3: Modal renderiza 3 secoes quando `status=success`: (a) **Proprietario** — nome, CPF/CNPJ mascarado, data da matricula, cartorio, data da ultima transacao; (b) **Contatos enriquecidos** — junta dados de `scraped_listings.nome_anunciante / telefone_anunciante / email_anunciante` do Epic 6 via `fn_enriched_contacts_by_edificio(p_edificio_id)` (criar nesta story) com deduplicacao por telefone; (c) **Proxima acao** — botao "Captar lead" que abre `CaptarLeadModal` (Story 6.5) pre-preenchido com nome do proprietario e melhor contato.
- [x] AC4: Estados de erro:
  - `status=not_found` -> mensagem amigavel "Cartorio nao localizou matricula para este endereco" + link para GeoSampa.
  - `status=failed` -> "Erro ao consultar cartorio. Tente novamente." + botao retry.
  - `forbidden` -> "Limite de 30 consultas/hora atingido. Tente novamente apos HH:MM."
  - `budget_exceeded` -> "Orcamento mensal de consultas atingido (R$X de R$Y). Aguarde proximo mes ou ajuste em Configuracoes."
- [x] AC5: LGPD UX: rodape fixo do modal com micro-copy "Dados obtidos de cartorios publicos via ARISP. Uso restrito a prospeccao profissional. [Esquecer dados]". Link dispara `POST /api/owners/:id/forget` com confirmacao; sucesso remove entrada da tabela e retira do cache.
- [x] AC6: Historico de lookups: nova aba "Proprietarios" na tela `mais/` lista os ultimos 50 lookups do consultor com filtro por edificio e status. Reusa componentes de tabela existentes.
- [x] AC7: Dashboard de consumo: no mesmo componente do AC6, header com cards "Consultas no mes", "Custo acumulado", "Cache hits / misses". Dados via RPC `fn_owner_lookup_stats(p_consultant_id)` (criada nesta story) agregando `owner_lookups`.
- [x] AC8: Analytics leve: disparar `intelligence_feed` tipo `owner_lookup_aberto` quando modal abre (ajuda measuring adoption). Nao disparar em cache hits silenciosos se flag `?quiet=1` estiver presente.
- [x] AC9: Acessibilidade: modal segue pattern dos modais existentes (focus trap, escape fecha, aria-label). Botao "Quem e o dono?" com `aria-label="Consultar proprietario via cartorio (custa R$0,28)"`.

## Technical Notes

- **Composicao de dados:** endpoint `/api/owners/lookup` (Story 6.6) ja retorna o proprietario. O modal complementa com `fn_enriched_contacts_by_edificio` chamado via `createClient().rpc(...)` no componente server-side se possivel, ou via hook dedicado `useEnrichedContactsByEdificio`.
- **Rate / budget UI:** response do endpoint inclui `rate_remaining, rate_reset_at, budget_used, budget_limit`; modal lê os headers/payload para mostrar estado sem nova request.
- **Cache freshness:** badge "Dados de cache (X dias)" calcula `(now() - created_at).days` baseado no payload retornado (Story 6.6 AC6).
- **Error taxonomy:** status mapping codificado em `app/src/lib/owner-lookup-errors.ts` para ser testavel.

## Dev Notes

- **AC1 / `tipo_residencial` (adaptação consciente):** o schema não tem `edificio.tipo_residencial` boolean — o dado real é `edificios_qualificacoes.tipologia`. `isResidencialTipologia` oculta o botão para `comercial`/`outro` e mostra para residencial vertical/horizontal, misto e tipologia desconhecida (Moema é majoritariamente residencial). Estados de erro/limite (AC1) vivem no modal, não no botão.
- **AC2 / lookup como `useQuery`:** o linter React Compiler (`react-hooks/set-state-in-effect` e `refs`) rejeitou o padrão mutation-em-effect; o lookup do modal virou `useQuery` com `postOwnerLookup` compartilhado (retry manual = `refetch()`). "Não chama API" em cache hit é garantido pelo endpoint da 6.6 (cache-first, custo 0), não pelo client.
- **AC6 / "aba" em `mais/`:** a tela `mais/` é um grid de rotas (não host de abas) — seguiu-se o padrão da call list 10.1: entrada no menu → rota `/proprietarios`.
- **AC7 / cache hits:** cache hit não insere linha; a contagem vem de `cache_hit_count` incrementado na linha servida (`recordCacheHit`, best-effort; read-modify-write aceitável no MVP single-consultant).
- **AC3c / Captar lead:** habilitado só quando há anúncio vinculado (a `useCaptarLead` exige `listing.id` real para o vínculo scraped_listing→lead); sem anúncio, hint orienta captação manual. Prefill usa nome do proprietário + melhor contato (telefone > whatsapp > email).
- **AC8 / quiet:** sem prop explícita, `?quiet=1` é lido da URL no botão; o evento é disparado após o resultado (única forma de saber `cache_hit`).
- **Pendência de deploy:** migrations 023 e 024 precisam ser aplicadas juntas no Supabase SQL Editor.

## Risks

| Risco | Mitigacao |
|---|---|
| Consultora dispara por engano e gasta budget | AC1 tooltip mostra custo + confirmacao implicita via modal (segundo clique). Cache elimina double-charge. |
| Dados do proprietario vazarem na URL | Modal nao navega, state mantido em memoria; logs mascaram CPF. |
| UX confusa com 3 secoes muito densas | Cada secao colapsavel; default: Proprietario expandido, Contatos colapsado. |
