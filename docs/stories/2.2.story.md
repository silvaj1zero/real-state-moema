# Story 2.2: Funil de Vendas RE/MAX (5 Etapas)

**Epic:** 2 — Leads, Funil de Vendas & Metodologia RE/MAX
**Status:** Done
**Priority:** Must
**Estimated Complexity:** XL

## User Story

Como consultora RE/MAX,
Quero visualizar e gerenciar meu pipeline de vendas em 5 etapas com transições rastreadas,
Para que eu controle cada lead do primeiro contato até a venda com registro completo de movimentações.

## Acceptance Criteria

- [ ] AC1: Tela "Funil de Vendas" acessível via tab "Funil" na barra inferior (ícone funil, badge com total de leads ativos). Mobile usa TABS horizontais com scroll (NOT Kanban horizontal). 5 tabs: Contato, V1, V2, Exclusividade, Venda. Cada tab exibe contador numérico entre parênteses. Tab ativa: texto azul bold #003DA5 + underline 3px azul. Tab inativa: texto cinza #6B7280. Swipe horizontal na área de conteúdo troca de tab com animação slide. Componente `FunnelTabs` (seção 3.4 do UX spec).
- [ ] AC2: Dentro de cada tab, leads agrupados por urgência: "Hoje" (leads com ação agendada para hoje), "Esta semana" (próximos 7 dias), "Sem ação" (leads parados >3 dias, fundo rosa claro #FEF2F2), "Todos" (restante). Query: `SELECT * FROM leads WHERE consultant_id = auth.uid() AND etapa_funil = $1 ORDER BY etapa_changed_at DESC`. Leads exibidos como `LeadCard` variant="full" (~120px altura) com nome (16px bold), edifício+unidade (14px cinza), tags (FISBO vermelho, FROG azul, padrão), próxima ação (cor por urgência: verde >2d, amarelo 1-2d, vermelho atrasado), botões inline (Ligar, Agendar, ...).
- [ ] AC3: Transição de etapa via swipe right no lead card OU tap no botão ">" do card. Abre modal de confirmação com: título "Mover [Nome] — [EtapaAtual] → [EtapaSeguinte]?", campo data (default hoje, editável via calendar picker), campo nota/observação (textarea, obrigatório). Ao confirmar: INSERT em `funnel_transitions` (lead_id, consultant_id, from_etapa, to_etapa, is_retrocesso=false, observacao) + UPDATE em `leads` SET `etapa_funil = nova_etapa`. Trigger `fn_update_lead_etapa_timestamp` atualiza `etapa_changed_at` automaticamente. Animação: card desliza para direita e desaparece. Toast: "[Nome] movido para [Etapa]" com haptic feedback.
- [ ] AC4: Retrocesso de etapa com 3 guardrails (Princípio PV). Swipe left no lead card → menu secundário → opção "Retroceder". Modal de retrocesso com campo obrigatório "Justificativa" (textarea, mínimo 10 caracteres). INSERT em `funnel_transitions` com `is_retrocesso = true` e `justificativa` preenchida. Trigger DB `fn_validate_funnel_transition` rejeita INSERT se `is_retrocesso = true AND justificativa IS NULL`. Guardrail 1: justificativa obrigatória. Guardrail 2: alerta visual permanente no lead card (ícone warning amarelo #EAB308, tooltip "Retrocedido em [data]: [justificativa]"). Guardrail 3: retrocesso contabilizado no diagnóstico (Story 2.8).
- [ ] AC5: Timeline/histórico de transições acessível ao tocar no lead card → tela de detalhe → seção "Histórico". Query: `SELECT * FROM funnel_transitions WHERE lead_id = $1 ORDER BY created_at DESC`. Exibe lista cronológica vertical com: data, de→para, observação, badge "Retrocesso" (vermelho) se aplicável. Cada entrada com ícone e cor da etapa destino.
- [ ] AC6: Filtros acessíveis via ícone filtro no header. Opções: por edifício (dropdown com busca), por origem (chips: FISBO, Informante, FROG, Digital), por data de criação (date range picker), por urgência (chips: Hoje, Esta semana, Sem ação). Filtros aplicados mostram badge numérico no ícone de filtro. Botão "Limpar filtros" visível quando filtros ativos.
- [ ] AC7: Contadores por etapa exibidos nas tabs e também como resumo no topo: "Total: N leads ativos". Query: `SELECT etapa_funil, COUNT(*) FROM leads WHERE consultant_id = auth.uid() AND etapa_funil NOT IN ('venda','perdido') GROUP BY etapa_funil`. Pull-to-refresh atualiza lista e contadores. TanStack Query key: `['leads', 'funnel', etapa_funil, filters]`.
- [ ] AC8: Desktop/tablet exibe Kanban horizontal (5 colunas lado a lado) com drag-and-drop entre colunas usando mesma lógica de transição (modal obrigatório). Media query breakpoint: `min-width: 768px` para Kanban, abaixo disso tabs mobile. Cores das colunas: Contato cinza #6B7280, V1 azul #003DA5, V2 azul escuro #001D4A, Exclusividade dourado #D97706, Venda verde #22C55E.

## Technical Notes

- **Database:** SELECT/UPDATE em `leads` (etapa_funil, etapa_changed_at). INSERT em `funnel_transitions` (lead_id, consultant_id, from_etapa, to_etapa, is_retrocesso, justificativa, observacao). Trigger `fn_validate_funnel_transition` bloqueia retrocesso sem justificativa no nível DB. Trigger `fn_update_lead_etapa_timestamp` atualiza `etapa_changed_at`.
- **API:** Supabase client SDK para SELECT com filtros dinâmicos. TanStack Query com `queryKey: ['leads', 'funnel', stage, filters]`. Mutations com optimistic update (remove card da tab atual, adiciona na nova tab antes da resposta do server).
- **Components:** FunnelTabs (tabs com swipe gesture), LeadCard (variant full com swipe gestures), TransitionModal (confirmação de avanço), RetrocessoModal (com justificativa obrigatória), FunnelTimeline (histórico), FunnelFilters (overlay com chips), FunnelKanban (desktop only, drag-and-drop).
- **Gestures:** react-swipeable ou framer-motion para swipe right (avançar) e swipe left (menu retroceder/arquivar). Threshold de 100px para confirmar swipe.
- **Offline:** Transições armazenáveis offline via mutation queue (Story 1.6). Ao sync, valida ordem cronológica.

## Notes

- **Princípio PV — Retrocesso com 3 guardrails:** Justificativa obrigatória (DB-level via trigger), alerta visual permanente no card, contabilizado no diagnóstico de gargalos (Story 2.8). Retrocesso não é proibido, mas é RASTREADO e VISÍVEL.
- **Princípio PV — Tabs mobile, NOT Kanban horizontal:** Mobile deve usar tabs (uma etapa por vez) por limitação de espaço e usabilidade com uma mão. Kanban é apenas para desktop (>=768px).
- A etapa "perdido" existe no enum `etapa_funil` mas não tem tab dedicada — leads perdidos ficam em filtro especial ou no diagnóstico.
- Transition modal com data+observação obrigatórias garante rastreabilidade completa para o diagnóstico (Story 2.8).
