# Story 3.7: Feed de Inteligência e Alertas

**Epic:** 3 — Inteligência, ACM & Agentes de Automação
**Status:** Ready
**Priority:** Must
**Estimated Complexity:** L

## User Story

Como consultora RE/MAX,
Quero receber um feed cronológico de eventos relevantes com alertas no mapa e notificações push para alta prioridade,
Para que eu identifique oportunidades rapidamente e nunca perca um FISBO novo, uma redução de preço ou um lead parado.

## Acceptance Criteria

- [ ] AC1: Feed cronológico de eventos acessível via tab "Inteligência" no menu principal. Lista de cards ordenada por `intelligence_feed.created_at DESC` com infinite scroll (20 items por página, TanStack Query `useInfiniteQuery`). Cada card exibe: ícone por tipo (ver AC2), título (TEXT bold 16px), descrição (TEXT regular 14px, max 2 linhas com ellipsis), timestamp relativo ("há 2h", "ontem", "3 dias atrás"), badge de prioridade (alta: vermelho #EF4444, média: amarelo #F59E0B, baixa: cinza #9CA3AF). Card clicável — tap navega para contexto (edifício no mapa, lead no funil, listing detail). Query: `SELECT * FROM intelligence_feed WHERE consultant_id = auth.uid() ORDER BY created_at DESC`.
- [ ] AC2: Tipos de evento com ícone e cor distintos. Enum `tipo_feed` com representação visual: (a) `novo_fisbo` — ícone casa+pessoa, cor verde #22C55E, prioridade default `alta`, (b) `reducao_preco` — ícone seta para baixo, cor laranja #F97316, prioridade default `media`, (c) `ex_imobiliaria_fisbo` — ícone troca, cor vermelho #EF4444, prioridade default `alta`, (d) `novo_raio_desbloqueado` — ícone alvo expandindo, cor azul #3B82F6, prioridade default `media`, (e) `lead_parado` — ícone relógio, cor amarelo #F59E0B, prioridade default `media`, (f) `agendamento_proximo` — ícone calendário, cor azul #3B82F6, prioridade default `media`, (g) `seed_completo` — ícone banco de dados, cor cinza #6B7280, prioridade default `baixa`, (h) `sync_completo` — ícone sincronizar, cor cinza #6B7280, prioridade default `baixa`.
- [ ] AC3: Pins de alerta no mapa. Eventos com `coordinates` NOT NULL geram pin overlay no mapa (Story 1.2). Pins de alerta: ícone pulsante (animação pulse CSS, 2s infinite) com cor do tipo de evento (AC2). Ao tap no pin de alerta: popover com título + descrição + botão "Ver detalhes" (navega para feed item). Pins de alerta filtráveis: toggle "Mostrar alertas" no mapa (default ON). Pins desaparecem quando evento marcado como lido. Query: `SELECT * FROM intelligence_feed WHERE consultant_id = auth.uid() AND is_read = false AND coordinates IS NOT NULL`.
- [ ] AC4: Filtros no feed. Bar de filtros no topo do feed: (a) por tipo — chips multi-select com ícones (AC2), default todos selecionados, (b) por prioridade — chips "Alta" / "Média" / "Baixa", default "Alta" + "Média", (c) por período — "Hoje" / "Semana" / "Mês" / "Todos", default "Semana", (d) por status leitura — toggle "Não lidos" / "Todos", default "Não lidos". Filtros aplicados como WHERE clause na query. Contagem por tipo exibida nos chips: "FISBO (3)".
- [ ] AC5: Badge de contagem no app. Ícone do tab "Inteligência" exibe badge numérico vermelho com contagem de não lidos: `SELECT COUNT(*) FROM intelligence_feed WHERE consultant_id = auth.uid() AND is_read = false`. Badge atualizado em tempo real via Supabase Realtime subscription no canal `intelligence_feed`. Badge desaparece quando todos lidos. Se contagem > 99, exibe "99+".
- [ ] AC6: Push notification para eventos de alta prioridade. Quando INSERT em `intelligence_feed` com `prioridade = 'alta'`: (a) Supabase database webhook dispara Edge Function `send-push-notification`, (b) Edge Function envia push via Web Push API (Service Worker registrado na Story 1.1 ou 2.6), (c) Payload: `{title: evento.titulo, body: evento.descricao, icon: '/icons/alert-[tipo].png', data: {feed_id: evento.id}}`, (d) Tap na notificação abre app no feed item específico. Push APENAS para prioridade `alta` — média e baixa são passive (apenas feed). UPDATE `intelligence_feed` SET `is_push_sent = true` após envio.
- [ ] AC7: Resumo matinal. Edge Function `morning-summary` executada via cron diário às 07:00 BRT. Compila resumo das últimas 24h: "Bom dia, Luciana! Ontem: [N] novos FISBOs, [N] reduções de preço, [N] leads parados. [Destaques: top 3 eventos por prioridade]." INSERT em `intelligence_feed` com tipo customizado ou push notification com corpo do resumo. Se nenhum evento relevante nas últimas 24h: "Bom dia! Sem novidades — um bom dia para prospectar! 🏠".
- [ ] AC8: Marcar como lido individual e em massa. (a) Swipe left no card do feed: botão "Lido" (azul). UPDATE `intelligence_feed` SET `is_read = true` WHERE `id = $1`. (b) Botão "Marcar todos como lidos" no topo do feed: UPDATE `intelligence_feed` SET `is_read = true` WHERE `consultant_id = auth.uid() AND is_read = false`. Animação: card desliza para esquerda e desaparece (se filtro = "Não lidos") ou muda opacidade para 0.6. TanStack Query mutation com optimistic update.
- [ ] AC9: Geração automática de evento `lead_parado`. Edge Function `check-stale-leads` executada via cron diário às 08:00 BRT. Query: `SELECT * FROM leads WHERE etapa_funil NOT IN ('concluido', 'perdido') AND updated_at < NOW() - INTERVAL '7 days'`. Para cada lead parado: INSERT em `intelligence_feed` com tipo `lead_parado`, prioridade `media`, título "Lead parado: [nome] — [etapa] há [N] dias", descrição "Última atividade: [data]. Considere fazer follow-up.", lead_id FK, coordinates herdado do edifício. Apenas 1 alerta por lead por semana (deduplica por `lead_id` + `tipo` + `created_at > NOW() - INTERVAL '7 days'`).

## Technical Notes

- **Database:** SELECT/INSERT/UPDATE em `intelligence_feed` (id, consultant_id, tipo, prioridade, titulo, descricao, coordinates, edificio_id, lead_id, scraped_listing_id, metadata JSONB, is_read, is_push_sent, created_at). Enum `tipo_feed`, `prioridade_feed`. Indexes: `idx_intelligence_feed_created` em `(created_at DESC)`, `idx_intelligence_feed_unread` em `(consultant_id) WHERE is_read = false`, GIST em `coordinates`.
- **Realtime:** Supabase Realtime subscription em `intelligence_feed` para badge e novos eventos. Canal: `intelligence_feed:consultant_id=eq.{uid}`.
- **Push:** Web Push API via Service Worker. VAPID keys em Supabase secrets. Push subscription salva em tabela `push_subscriptions` (consultant_id, endpoint, keys_p256dh, keys_auth).
- **Cron:** `pg_cron` ou external cron para: morning summary (07:00), stale leads check (08:00).
- **API:** Supabase client SDK. TanStack Query keys: `['feed', filters]`, `['feed', 'count']`. `useInfiniteQuery` para pagination. Mutations para marcar lido.
- **Components:** FeedScreen (tela principal), FeedCard (card de evento), FeedFilters (bar de filtros), FeedBadge (badge no tab), MapAlertPin (pin pulsante no mapa), FeedSwipeActions (swipe para lido), MorningSummaryCard (resumo matinal).

## Notes

- **Princípio PV — Automação sugere, humano decide:** Feed apresenta inteligência, consultora decide como agir. Eventos são informativos, não prescritivos.
- **Princípio PV — Graceful degradation:** Se scraping (3.4) não está ativo, feed funciona com eventos de outras fontes (lead_parado, agendamento_proximo, raio_desbloqueado). Feed nunca está "vazio" em uso normal.
- Push notifications APENAS para alta prioridade para evitar notification fatigue. Média e baixa são passive — consultora consulta quando quiser.
- Resumo matinal é touchpoint diário — mantém consultora engajada e informada sobre oportunidades.
- Badge no tab é o principal driver de engagement — número vermelho chama atenção e cria urgência.
- Evento `lead_parado` é gerado por cron, não por trigger — verifica leads sem atividade há 7+ dias. Deduplica para evitar spam.
