# Story 2.3: Gestão de Informantes e Marketing de Gentileza

**Epic:** 2 — Leads, Funil de Vendas & Metodologia RE/MAX
**Status:** Done
**Priority:** Must
**Estimated Complexity:** XL

## User Story

Como consultora RE/MAX,
Quero cadastrar informantes (zeladores, porteiros, síndicos) vinculados a edifícios e registrar ações de gentileza,
Para que eu construa relacionamentos com quem gera indicações e rastreie o retorno do investimento.

## Acceptance Criteria

### Fase A — Cadastro + Gentileza

- [x] AC1: Cadastro de informante acessível via: (a) card do edifício → seção "Informantes" → botão "+ Informante", (b) FAB long-press → menu radial → "Informante". Formulário com campos: nome (input 48px, obrigatório), função (chips single-select: Zelador / Porteiro / Gerente Predial / Comerciante / Síndico / Outro — enum `funcao_informante`), telefone (input com máscara BR, criptografado via `pgp_sym_encrypt` em `informantes.telefone_encrypted` BYTEA), qualidade da relação (chips: Frio / Morno / Quente — CHECK constraint em `informantes.qualidade_relacao`), notas (textarea). INSERT em `informantes` com `consultant_id = auth.uid()`.
- [x] AC2: Vinculação informante↔edifícios via multi-select. Ao cadastrar, campo "Edifícios" com busca e seleção múltipla (chips com X para remover). INSERT em `informantes_edificios` (informante_id, edificio_id) para cada edifício selecionado. Um informante pode estar vinculado a N edifícios (ex: zelador que cuida de 3 prédios). UNIQUE constraint em `(informante_id, edificio_id)`.
- [x] AC3: Registro de ações de gentileza via botão "Registrar Gentileza" no detalhe do informante. Formulário: tipo (chips: Café / Brinde / Agradecimento Escrito / Presente / Outro — CHECK constraint em `acoes_gentileza.tipo`), descrição (textarea), valor R$ (input numérico, default 0), data (date picker, default hoje). INSERT em `acoes_gentileza` (informante_id, consultant_id, tipo, descricao, valor, data_acao). Após salvar, `informantes.total_investido_gentileza` atualizado via SUM query ou trigger.
- [x] AC4: Histórico de gentilezas no detalhe do informante. Lista cronológica com: data, tipo (ícone), descrição, valor. Query: `SELECT * FROM acoes_gentileza WHERE informante_id = $1 ORDER BY data_acao DESC`. Total investido exibido no topo: "Total investido: R$ [valor]" (font 18px bold, verde #22C55E).

### Fase B — 5% Comissão + Lembretes

- [x] AC5: Tracking de leads originados por informante. No detalhe do informante, seção "Leads indicados" com lista de leads onde `leads.informante_id = informante.id`. Contagem total + status de cada lead no funil. Cálculo de comissão 5% (Regra de Ouro): quando lead chega a "Venda", sistema calcula 5% do valor da venda e exibe como "Comissão devida: R$ [valor]". Atualiza `informantes.comissao_devida`. **Sistema sugere, humano confirma** — botão "Confirmar pagamento" atualiza `informantes.comissao_paga` (nunca auto-pagar). Badge "Comissão pendente" (dourado #D97706) no card do informante se comissao_devida > comissao_paga.
- [x] AC6: Lembretes de contato. Se último `acoes_gentileza.data_acao` > 30 dias, badge "Há [N] dias sem contato" (vermelho #DC1431) no card do informante. Feed de inteligência (Story 3.7) gera alerta "Lembrete: visitar [Informante] no [Edifício]". Lista de informantes ordenável por "último contato" (ASC = mais negligenciados primeiro).

### Fase C — Dashboard

- [x] AC7: Seção "Informantes" no Dashboard de KPIs (Story 2.9) com: total de informantes ativos, total de indicações (leads com informante_id not null), total investido em gentileza, comissão potencial (sum de comissao_devida), ROI (leads convertidos / total investido). Cards KPI 165x80px, número 24px bold, label 12px cinza. Empty state: "Cadastre seu primeiro informante para começar a rastrear indicações!".
- [x] AC8: Lista completa de informantes acessível via tab "Mais" → "Informantes". Exibe card por informante: nome, função (chip), edifícios vinculados (badges), qualidade da relação (cor: Frio=azul, Morno=amarelo, Quente=vermelho), leads indicados (contagem), total investido. Ordenável por: nome, qualidade, total investido, último contato. Busca por nome.

## Technical Notes

- **Database:** INSERT em `informantes` (consultant_id, nome, funcao, telefone_encrypted, qualidade_relacao, notas, total_investido_gentileza, comissao_devida, comissao_paga). INSERT em `informantes_edificios` (informante_id, edificio_id). INSERT em `acoes_gentileza` (informante_id, consultant_id, tipo, descricao, valor, data_acao). Criptografia: `pgp_sym_encrypt(telefone, current_setting('app.encryption_key'))`.
- **API:** Supabase client SDK. TanStack Query keys: `['informantes', consultant_id]`, `['informantes', informante_id, 'gentilezas']`, `['informantes', informante_id, 'leads']`. Mutation com optimistic update no card do edifício (contagem informantes).
- **Components:** InformanteForm (cadastro), InformanteCard (lista), InformanteDetail (detalhe com histórico), GentilezaForm (registro), GentilezaTimeline (histórico), ComissaoTracker (cálculo 5%), InformanteList (lista completa com filtros).
- **Implementação incremental:** Fase A é o MVP (cadastro+gentileza). Fase B adiciona tracking financeiro e lembretes. Fase C integra com dashboard. Cada fase pode ser entregue e testada independentemente.

## Notes

- **Princípio PV — Comissão: sistema sugere, humano confirma.** O sistema calcula 5% automaticamente ao lead converter, mas NUNCA marca como paga automaticamente. Sempre requer confirmação manual da consultora. Isso é crítico por envolver dinheiro.
- **LGPD (NFR-003):** Telefone do informante DEVE ser criptografado via pgcrypto, igual ao lead.
- A vinculação N:M (informante↔edifícios) é essencial — um zelador pode cuidar de múltiplos prédios.
- Implementação incremental (Fase A→B→C) reduz risco e permite validação com a Luciana a cada fase.
