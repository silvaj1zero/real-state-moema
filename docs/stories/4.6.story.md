# Story 4.6: Preparação Multi-Tenant

**Epic:** 4 — Parcerias Ganha/Ganha & Escala
**Status:** Draft
**Priority:** Must
**Estimated Complexity:** XL

## User Story

Como administradora do sistema,
Quero que o sistema suporte múltiplos consultores com isolamento de dados via RLS,
Para que outros consultores RE/MAX possam usar a plataforma sem ver dados privados uns dos outros, mantendo edifícios como dados públicos compartilhados.

## Acceptance Criteria

- [ ] AC1: Ativação de Row Level Security (RLS) em TODAS as tabelas com `consultant_id`: `edificios_qualificacoes`, `leads`, `funnel_transitions`, `informantes`, `acoes_gentileza`, `referrals`, `comissoes`, `intelligence_feed`, `consultant_settings`, `safari_events`, `marketing_plans`. Para cada tabela: `ALTER TABLE [table] ENABLE ROW LEVEL SECURITY` + `CREATE POLICY "[table]_own" ON [table] FOR ALL USING (consultant_id = auth.uid())`. Policies já definidas na migration 004 (comentadas) — descomentar e aplicar.
- [ ] AC2: Exceção CRÍTICA — Edifícios são PÚBLICOS: tabela `edificios` NÃO recebe RLS. Todos os consultores veem todos os edifícios do mapa. Tabela `edificios_qualificacoes` tem RLS — cada consultor vê apenas SUAS qualificações (status_varredura, notas, etc.) de cada edifício. **ALERTA PV:** Essa separação edificios (público) / qualificacoes (privado) já está no design desde Epic 1.
- [ ] AC3: Exceção referrals: policy especial para `referrals` — consultor vê referrals onde `consultant_id = auth.uid()`. Não há necessidade de policy cruzada (sistema unilateral — parceiros não são users). Se no futuro parceiros forem users, adicionar policy `OR parceiro_user_id = auth.uid()`.
- [ ] AC4: Convite por email: tela "Convidar Consultor" (acessível para admin — role check). Formulário: email, nome, franquia. Ao submeter, Supabase Auth `signUp` com email + password temporário ou magic link. INSERT em `consultores` com dados básicos. INSERT em `consultant_settings` com defaults. Email enviado via Supabase Auth (template customizado com branding RE/MAX). Novo consultor ao fazer login entra com dados isolados por RLS.
- [ ] AC5: Dashboard admin (classificação **Could** — schema only): definir schema para tela administrativa: lista de consultores (nome, franquia, data cadastro, último login, contagem leads/comissões). Endpoints: `SELECT consultores.*, count(leads.id) FROM consultores LEFT JOIN leads...`. NÃO implementar UI nesta story — apenas garantir que queries são possíveis e schema suporta. Admin role via `consultores.role` (adicionar coluna TEXT DEFAULT 'consultant' CHECK IN ('consultant','admin')).
- [ ] AC6: Migração zero-downtime: script de migração que ativa RLS sem afetar o usuário existente (Luciana). Passos: (1) criar policies, (2) ativar RLS, (3) verificar que Luciana continua acessando todos os seus dados, (4) smoke test. Rollback: `ALTER TABLE [table] DISABLE ROW LEVEL SECURITY`. Migration file dedicado ou seção na migration 004. Executar em horário de baixo uso.
- [ ] AC7: Testes E2E de isolamento OBRIGATÓRIOS: test suite com 2 consultores fictícios (Consultor A e Consultor B). Cenários: (a) Consultor A cria lead — Consultor B NÃO vê, (b) Consultor A qualifica edifício — Consultor B NÃO vê qualificação mas VÊ edifício, (c) Consultor A cria referral — Consultor B NÃO vê, (d) Ambos veem mesmos edifícios base, (e) Consultor A cria comissão — Consultor B NÃO vê, (f) Admin vê tudo (futuro). Testes executados com Supabase service_role key para setup e anon/authenticated keys para assertions. **Falha em qualquer teste = bloqueio de merge.**
- [ ] AC8: Seed data de edifícios mantido como público: edifícios inseridos via Story 1.7 (seed) ou via cadastro (Story 1.3) permanecem visíveis para todos. Campo `edificios.is_seed` (BOOLEAN, existente ou a adicionar) marca edifícios de referência. Qualificações são SEMPRE privadas. Ao onboardar novo consultor, ele vê o mapa com todos os edifícios mas zero qualificações/leads/comissões.
- [ ] AC9: Verificação de integridade pós-ativação: query de validação que confirma: (a) todas as tabelas com `consultant_id` têm RLS ativo, (b) todas têm pelo menos 1 policy, (c) tabela `edificios` NÃO tem RLS, (d) contagem de registros por consultor está consistente. Script SQL executável manualmente ou via Edge Function.

## Technical Notes

- **Database:** ALTER TABLE + CREATE POLICY para 11 tabelas (listadas em AC1). Policies seguem padrão `USING (consultant_id = auth.uid())`. Migration 004 já tem SQL comentado — descomentar e aplicar. Adicionar coluna `consultores.role` (TEXT DEFAULT 'consultant' CHECK IN ('consultant','admin')).
- **Auth:** Supabase Auth com `auth.uid()` mapeado para `consultores.auth_user_id`. Garantir que todos os consultores têm `auth_user_id` preenchido. Magic link para convite (Supabase Auth built-in).
- **Testes:** Supabase test helpers ou Playwright E2E com 2 sessions autenticadas. Framework: Jest/Vitest para unit, Playwright para E2E. Fixtures: 2 consultores, 5 edifícios compartilhados, leads/qualificações privadas.
- **Rollback:** `ALTER TABLE [table] DISABLE ROW LEVEL SECURITY` remove isolamento sem perder dados. Policies podem ser dropadas com `DROP POLICY [name] ON [table]`.
- **Components:** InviteConsultorForm, AdminDashboard (schema only — Could).

## Dependencies

- Blocks: 4.5 (multi-tenant deve estar ativo antes de sync externo)

## Notes

- **ALERTA PV — Separação já no design:** A separação edifícios (públicos) vs qualificações (privadas) foi definida desde o Epic 1. O multi-tenant ativa o que já estava planejado. Não é refactoring — é ativação.
- **Princípio PV — Testes obrigatórios:** Isolamento de dados é crítico. Um vazamento de leads entre consultores é falha grave. Testes E2E de isolamento são gate obrigatório — sem aprovação, não faz merge.
- **Princípio PV — Zero-downtime:** Luciana já está usando o sistema. A ativação de RLS não pode derrubar o serviço nem bloquear o acesso dela. Migration cuidadosa com rollback preparado.
- Dashboard admin como Could é intencional — o foco do Epic 4 é habilitar multi-tenant, não construir admin completo. Schema ready, UI depois.
