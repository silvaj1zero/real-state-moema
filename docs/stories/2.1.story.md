# Story 2.1: Cadastro de Leads (Proprietários FISBO)

**Epic:** 2 — Leads, Funil de Vendas & Metodologia RE/MAX
**Status:** InProgress
**Priority:** Must
**Estimated Complexity:** L

## User Story

Como consultora RE/MAX,
Quero cadastrar proprietários vinculados a edifícios do mapa,
Para que eu tenha uma base estruturada de leads FISBO com perfil completo.

## Acceptance Criteria

- [ ] AC1: Botão "+ Lead" acessível a partir do card do edifício (Story 1.4, estado Half/Full). Botão outlined azul #003DA5, 36px altura, ícone pessoa+. Também acessível via FAB long-press → menu radial → "Lead".
- [x] AC2: Formulário de cadastro com campos obrigatórios: nome do proprietário (input 48px, auto-focus), unidade (apt/sala, input 48px), origem do lead (chips: Digital / Placa / Zelador / Indicação — 40px, single-select). INSERT em `leads` com `edificio_id` FK, `consultant_id = auth.uid()`.
- [x] AC3: Campos de perfil (opcionais, expansíveis via "Mais detalhes"): perfil psicográfico (chips: Investidor / Herdeiro / Mudança de cidade / Upgrade-Downgrade), o que valoriza (chips: Preço / Rapidez / Discrição / Segurança). Salvos em `leads.perfil_psicografico` e `leads.valoriza`.
- [x] AC4: Campos V1 estruturados (seção "Dados da V1" — colapsada por default, expandível): motivação real da venda (textarea, `leads.motivacao_venda`), prazo de urgência (chips: Imediato / 3 meses / 6 meses / Sem pressa — enum `prazo_urgencia`), upload de fotos V1 (máx 5 fotos, comprimidas <1MB cada, salvas em Supabase Storage bucket `v1-fotos`, paths em `leads.fotos_v1` TEXT[]).
- [x] AC5: Telefone e email criptografados via `pgcrypto` — `leads.telefone_encrypted` BYTEA e `leads.email_encrypted` BYTEA. Input com máscara telefone BR (11) XXXXX-XXXX. Exibição com `pgp_sym_decrypt` apenas para o consultor dono (RLS futuro, app-level agora).
- [x] AC6: Lead automaticamente vinculado ao edifício (`leads.edificio_id` FK) e herda coordenadas. Card do edifício (Story 1.4) exibe contagem de leads: "3 leads" com badge numérico. Campo `leads.fonte_frog` preenchível (enum: familia/relacionamentos/organizacoes/geografia) — integra com Story 2.4.
- [ ] AC7: Se origem = "Zelador" ou "Indicação", campo adicional para selecionar informante existente (`leads.informante_id` FK → `informantes`). Se informante não existe, link "Cadastrar informante" que abre Story 2.3. Informante selecionado atualiza tracking (futuro Story 2.3).
- [x] AC8: Lista de leads acessível via: (a) card do edifício → seção "Leads" (max 3 visíveis, "Ver todos (N)"), (b) busca global (busca por nome, endereço, unidade — query em `leads` com `ILIKE`), (c) tab "Funil" (Story 2.2).
- [x] AC9: Ao salvar lead, toast "Lead [Nome] cadastrado!" com haptic feedback. Lead entra automaticamente na etapa "Contato" do funil (`leads.etapa_funil = 'contato'`). Pin do edifício não muda de cor (cor = status de varredura, não de lead).

## Technical Notes

- **Database:** INSERT em `leads` (consultant_id, edificio_id, informante_id, nome, unidade, telefone_encrypted, email_encrypted, origem, fonte_frog, etapa_funil='contato', motivacao_venda, prazo_urgencia, fotos_v1, perfil_psicografico, valoriza, is_fisbo, notas). Criptografia: `pgp_sym_encrypt(telefone, current_setting('app.encryption_key'))`. Decriptografia: `pgp_sym_decrypt(telefone_encrypted, current_setting('app.encryption_key'))`.
- **Storage:** Bucket `v1-fotos` no Supabase Storage. Path: `{consultant_id}/{lead_id}/{filename}`. Policy: apenas consultant dono pode ler/escrever.
- **API:** Supabase client SDK para INSERT. TanStack Query mutation com optimistic update no card do edifício (contagem de leads). Query key: `['leads', edificio_id]`.
- **Components:** LeadForm (formulário com seções colapsáveis), ChipSelect (single/multi), PhotoUploader (compressão client-side via canvas), LeadList (no card do edifício), GlobalSearch (busca em leads).
- **Offline:** Lead cadastrável offline via mutation queue (Story 1.6). Fotos armazenadas em IndexedDB, sync em background via WiFi.

## Notes

- **Princípio PV — Campos V1 estruturados:** Gap identificado na auditoria PV e resolvido — motivação real, prazo urgência e fotos são CAMPOS, não texto livre genérico. Isso permite o diagnóstico de gargalos (Story 2.8) e prepara o Dossiê (Story 3.2).
- **LGPD (NFR-003):** Telefone e email DEVEM ser criptografados via pgcrypto. Nunca armazenar em texto puro.
- Fotos V1 comprimidas client-side antes do upload — máx 1MB por foto, 5 fotos por lead. Usar canvas.toBlob() com quality 0.7.
- Campo `is_fisbo` default false — será marcado pela Story 2.10 ou automaticamente pelo agente de scraping (Epic 3).
