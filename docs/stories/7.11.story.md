# Story 7.11: Detecção FISBO determinística via `publisherType` nativo (ZAP/VivaReal)

**Epic:** 7 — Inteligencia de Prospeccao Automatizada Multi-Fonte
**Wave:** A (refino de qualidade)
**Status:** Draft
**Priority:** Should (alto ROI — baixo esforço, ganho direto de qualidade de lead)
**Estimated Complexity:** S (3-5 pts)

## User Story

Como sistema de prospeccao,
Quero usar o campo nativo `publisherType` (OWNER/AGENCY/DEVELOPER) dos feeds de ZAP/VivaReal como sinal determinístico,
Para classificar anunciantes (FISBO vs imobiliária vs construtora) com precisão direta, em vez de depender só da heurística 4-signal quando a fonte já entrega o dado.

## Contexto

A pesquisa `docs/research/2026-06-15-fisbo-captacao-ingestao/report.md` (achado F1, confiança alta 3-0) confirmou que os scrapers de ZAP/VivaReal (ex.: Apify `haketa/zapimoveis-scraper`, `jungle_synthesizer/brazil-vivareal-zap-imoveis-scraper`) expõem nativamente `publisherType` ∈ {OWNER, AGENCY, DEVELOPER} + presença de CRECI, em 40+ campos com nome/telefone/email do publisher. ZAP e VivaReal compartilham o backend Glue API da OLX.

Hoje `classifyAdvertiser` (`app/src/lib/scrapers/classify-advertiser.ts`, ADR-EPIC7-004) usa só a heurística 4-signal (CNPJ, CRECI, DDD/telefone móvel, contagem de anúncios) — estimativa 70-85% de precisão. Onde a fonte entrega `publisherType`, temos um sinal **determinístico** que supera a heurística.

## Acceptance Criteria

- [ ] AC1: Schema `app/src/lib/schemas/epic7/advertisers.ts` ganha campo opcional `publisher_type` enum `['owner','agency','developer']` (nullable — fontes sem o campo, ex.: MercadoLivre, ficam null).
- [ ] AC2: `classifyAdvertiser` ganha um **passo 0 determinístico** (antes da heurística): se `publisher_type` presente → mapeia direto: `owner→for_sale_by_owner` (confidence 0.95), `agency→broker` (0.95), `developer→builder` (0.95), com novo `ClassificationSignal` (`publisher_type_owner`/`_agency`/`_developer`). Sem `publisher_type` → cai na heurística 4-signal atual (inalterada).
- [ ] AC3: O parser/ingestão de ZAP/VivaReal mapeia o campo nativo do feed → `publisher_type` (normalizar OWNER/AGENCY/DEVELOPER → lower-case enum). MercadoLivre permanece null (segue heurística).
- [ ] AC4: Telemetria: registrar a fonte da classificação (`deterministic_publisher_type` vs `heuristic_4signal`) em `classification_signals` para auditoria/medição de cobertura.
- [ ] AC5: Testes em `app/src/lib/scrapers/__tests__/classify-advertiser.test.ts`: cada `publisher_type` → classificação correta + confidence 0.95; ausência de `publisher_type` → heurística preservada (regressão dos casos atuais verde); conflito (publisher_type=owner mas CRECI presente) → publisher_type vence, com signal de conflito registrado.
- [ ] AC6: `docs/architecture/adrs/ADR-EPIC7-004` referenciado/anotado: o passo determinístico **precede** a heurística (não a remove — é fallback).

## Dependencies

- Depends on: Story 7.1 (schemas Zod), Story 7.3 (`classifyAdvertiser`).
- Relaciona: ingestão ZAP/VivaReal (Epic 6 busca paramétrica) — fonte do `publisherType`.

## Out of Scope

- Trocar a heurística 4-signal (mantida como fallback p/ fontes sem o campo).
- Enriquecimento de contato do proprietário (Story futura — padrão Captei, com LIA/opt-out).
- Crawler de ZAP/VivaReal em si (já existe via Epic 6).

## Technical Notes

- **Art. IV (No Invention):** o mapeamento OWNER/AGENCY/DEVELOPER vem do schema real dos actors (report F1) — não inventar valores.
- Conflito publisher_type vs CRECI: priorizar `publisher_type` (sinal da plataforma) mas **logar** o conflito (um "OWNER" com CRECI pode ser corretor anunciando como PF — material para auditoria).
- Caveat do report: campo no schema ≠ campo sempre preenchido; tratar null com graça (fallback heurística).
