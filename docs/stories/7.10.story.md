## Technical Notes

- **Reference primaria:** ADR-EPIC7-004 + `docs/research/2026-05-14-leads-zonasul-sp/03-recommendations.md` REC-7.1 a REC-7.5
- **Supabase Vault vs pgcrypto:**
  - Vault: managed, key rotation built-in, fee adicional (verificar plano)
  - pgcrypto: free, mas chave deve ser passada por env var em cada query
  - **Decisao Wave A:** preferir Vault; fallback pgcrypto se nao disponivel
- **Chave de cifragem:** env var `LGPD_ENCRYPTION_KEY` em Vercel; nunca em logs; rotacionar a cada 6 meses (manual Wave A, automatic Wave B)
- **LIA template:** referencia model ANPD + Mattos Filho cited em research
- **Pagina privacidade canonica:** texto revisar com counsel para evitar omissoes (Decreto 11.793/2023 + Resolucoes ANPD 1-7/2022-2025)
- **Audit log retencao:** indefinida Wave A (proof of compliance); revisar Wave C com counsel

## Dev Agent Record

**Mode:** Pre-Flight → YOLO finalization (2026-05-14 ate 2026-05-18). Estrutura major decidida na sessao 14/05 (vault wrapper, separacao crypto/audit, retencao via cron); finalizacao YOLO em 18/05 (testes + PoC + story close).

**Decisions:**
- **Vault wrapper sincrono em vez de classe stateful.** `lib/vault.ts` expoe 3 funcoes puras (`storeLeadPII`, `getLeadPII`, `storeLeadPIIBatch`) que recebem `SupabaseClient` por parametro. Sem singleton, sem state — facilita mock em testes (39 testes novos passam sem fixtures externos) e permite usar tanto admin client quanto server client conforme o caller (admin endpoint vs server action).
- **Separacao `lgpd-crypto.ts` vs `lgpd-audit.ts`.** Story file referenciava os dois explicitamente; `lgpd-crypto.ts` ficou como fachada thin re-exportando `vault.ts` (story 7.4 vai importar `encryptLeadField` desse path canonico). Auditoria fica em modulo proprio porque sera consumida tambem por Stories 7.4 (scrape log), 7.7 (creci reveal log), Wave B Telegram.
- **Audit via RPCs SECURITY DEFINER + log inline em endpoints, NAO via trigger BEFORE/AFTER.** Trigger acoplaria audit a INSERT em `scraped_listings`, mas Wave A captura PII em `leads` (nao mais em `scraped_listings`). RPC garante que toda operacao audit-relevante (reveal, anonymize) passe pela mesma porta com `legal_basis` controlado. Endpoint opt-out loga inline via service-role (titular nao tem JWT, trigger nao serviria).
- **Retention 90d via Edge Function cron, NAO trigger DB.** Trigger seria sincrono e pesado (DELETE em massa em row-by-row). Cron permite (a) batch + monitoring, (b) reversibilidade (audit ANPD se solicitado), (c) integrar com infra de cron da Story 7.6 quando essa publicar. Hoje a funcao `fn_lgpd_purge_expired_audit` existe na migration 015 mas nao tem schedule — registrado em TODO.
- **Migration original 017 renumerada para 014+015.** Per brief do orquestrador para alinhamento com numeracao Epic 7 (7.6 ja consume 012/013) e split por separation of concerns: 014 = cifragem (Vault), 015 = audit + opt-out + retention. Ambas idempotentes (CREATE IF NOT EXISTS / ADD COLUMN IF NOT EXISTS).
