# Story 3.8: Enriquecimento de Leads

**Epic:** 3 — Inteligência, ACM & Agentes de Automação
**Status:** Done
**Priority:** Should
**Estimated Complexity:** M

## User Story

Como consultora RE/MAX,
Quero enriquecer um lead sob demanda com inteligência de mercado ao clicar em "Buscar inteligência",
Para que eu tenha dados acionáveis sobre anúncios, valor estimado e potencial FISBO antes de abordar o proprietário.

## Acceptance Criteria

- [x] AC1: Botão "Buscar inteligência" no card do lead. Botão outlined azul #003DA5, 36px altura, ícone lupa+cérebro. Visível apenas em leads com `edificio_id` NOT NULL (precisa de localização geográfica). Ao clicar, inicia processo de enriquecimento sob demanda. Loading state: botão muda para spinner + "Buscando..." (disabled durante execução). Tempo máximo: 10s timeout com fallback graceful.
- [x] AC2: **Sob demanda ONLY no MVP.** Enriquecimento NÃO é automático — APENAS quando consultora clica no botão. Sem cron, sem trigger automático, sem enriquecimento em background. Princípio: consultora controla o gasto de API calls e decide quando precisa de inteligência extra. Futuro: pode tornar-se automático para leads em etapas avançadas do funil.
- [x] AC3: Busca de anúncios no entorno do lead. Query: `SELECT * FROM scraped_listings WHERE is_active = true AND ST_DWithin(coordinates, (SELECT coordinates FROM edificios WHERE id = lead.edificio_id), 100) ORDER BY ST_Distance(...) LIMIT 10`. Resultados exibidos em seção "Anúncios no entorno" no card do lead: lista de cards compactos com endereço (truncado), preço (R$ formatado), área m², portal (chip colorido), tempo no mercado (badge). Se `scraped_listing.matched_edificio_id = lead.edificio_id`: highlight "Mesmo edifício" com badge verde. Se zero resultados: mensagem "Nenhum anúncio ativo encontrado no raio de 100m."
- [x] AC4: Estimativa de valor por m² para a micro-região do lead. Calcula usando `fn_comparaveis_no_raio(lat, lng, consultant_id, 500)` da Story 3.1. Resultado exibido em card de resumo: "Estimativa região: R$ [media]/m² (baseado em [N] comparáveis)". Se `is_venda_real` disponível, exibe separado: "Anúncio: R$ X/m² | Venda real: R$ Y/m²". Se nenhum comparável: "Sem dados de mercado — cadastre comparáveis na ACM para estimativa." com link para Story 3.1. Card com fundo gradient sutil azul→branco.
- [x] AC5: Indicador "Potencial FISBO". Algoritmo de scoring baseado em sinais disponíveis: (a) +30 pontos se `scraped_listing` com `is_fisbo = true` no mesmo edifício, (b) +20 pontos se `scraped_listing` com `tipo_anunciante = 'proprietario'` em raio 50m, (c) +15 pontos se edifício com `total_units > 20` (prédios grandes = mais chances de FISBO), (d) +15 pontos se transição ex-imobiliária→FISBO detectada no entorno (Story 3.6), (e) +10 pontos se tempo de mercado > 90 dias em listing próximo, (f) +10 pontos se redução de preço recente em listing próximo. Score total: 0-100. Exibido como badge no card do lead: "Potencial FISBO: [score]%" com cores: verde >= 60%, amarelo 30-59%, cinza < 30%. Tooltip com breakdown dos sinais que compõem o score.
- [x] AC6: Resultados do enriquecimento salvos no lead. UPDATE `leads` SET `enrichment_data = JSONB` com snapshot dos resultados: `{enriched_at, anuncios_entorno: [{id, preco, area_m2, portal, is_fisbo}], estimativa_m2: {media, mediana, total_comparaveis}, fisbo_score: {score, sinais: [...]}, scraped_match_id}`. Campo `leads.enrichment_data JSONB` (novo campo). Dados exibidos no card do lead em seção colapsável "Inteligência" (expandida por default após enriquecimento). Timestamp: "Atualizado há [N] dias" — botão "Atualizar" para re-enriquecer.
- [x] AC7: Resultado exibido em seção "Inteligência" no card expandido do lead. Layout: (a) Card "Estimativa de mercado" (AC4) — topo, destaque, (b) Badge "Potencial FISBO" (AC5) — ao lado do nome do lead, (c) Lista "Anúncios no entorno" (AC3) — seção colapsável, max 5 visíveis + "Ver todos (N)", (d) Metadata: "Última busca: [data] — [N] anúncios, [N] comparáveis". Seção visível apenas se `leads.enrichment_data` IS NOT NULL. Se null, exibe botão "Buscar inteligência" (AC1) centralizado na seção.

## Technical Notes

- **Database:** UPDATE em `leads` SET `enrichment_data JSONB`. SELECT em `scraped_listings` com PostGIS `ST_DWithin(100m)`. Function `fn_comparaveis_no_raio` para estimativa m². Novo campo `leads.enrichment_data JSONB` (ALTER TABLE).
- **FISBO Score:** Calculado client-side ou em Edge Function. Inputs: scraped_listings match, edificio attributes, intelligence_feed events. Output: score 0-100 + array de sinais.
- **API:** Supabase client SDK. Enriquecimento via single RPC call ou multiple parallel queries (scraped_listings + comparaveis + intelligence_feed). TanStack Query key: `['lead-enrichment', lead_id]`. Mutation para trigger re-enrichment.
- **Components:** LeadEnrichmentButton (botão "Buscar inteligência"), LeadIntelligenceSection (seção no card), MarketEstimateCard (estimativa m²), FisboScoreBadge (badge potencial), NearbyListingsCard (anúncios entorno), EnrichmentTimestamp (metadata atualização).
- **Performance:** Timeout 10s para todo o processo de enriquecimento. Queries paralelas (Promise.all): scraped_listings + comparaveis + intelligence_feed. Cache: resultado salvo em `enrichment_data` — re-fetch apenas quando consultora clica "Atualizar".

## Notes

- **Princípio PV — Enriquecimento sob demanda ONLY no MVP:** Consultora controla quando busca inteligência. Sem automação, sem gasto de API calls em background. Botão explícito = controle explícito.
- **Princípio PV — Graceful degradation:** Se scraped_listings vazio (sem scraping ativo), busca retorna zero anúncios mas estimativa m² ainda funciona com comparáveis manuais da ACM. Se ACM vazia, mensagem sugere cadastrar comparáveis. Sistema NUNCA falha — sempre exibe o que tem disponível.
- **Princípio PV — Scraping é acelerador, não dependência:** FISBO score funciona com dados parciais. Se apenas 1 sinal disponível, score reflete isso. Score 0% é informação válida ("sem sinais de FISBO detectados").
- FISBO score é heurístico — não é predição. Serve como priorização para a consultora decidir onde investir tempo.
- Resultados salvos em JSONB permitem snapshot point-in-time — consultora pode ver como dados mudaram entre enriquecimentos.
- Timeout de 10s com fallback garante que UI nunca fica travada — se API lenta, retorna resultados parciais disponíveis.
